## AI教育ケース #1

### 症例
テーマが「お任せ（omakase）」で始まったインタビューで、キャラが会話中盤以降も同じ水準の「広がりを求める質問」を繰り返し、一段深い掘り下げに進めない状態が起きうる。

具体的には「最近、お客様に喜んでもらえたことはありますか」「印象に残っているお客様を教えてください」といった入口質問が中盤以降も続き、事業者が「また同じような質問だな」と感じて答えが浅くなっていく。

### 何が弱かったか
`buildInterviewQualityContext`（`lib/ai-quality.ts`）は「中盤（3回以上）」と「終盤（6回以上）」にそれぞれ進め方の指示を出している。しかし「中盤の進め方」の内容は「判断基準・こだわり・お客様の反応のどれかを1つだけ掘れ」という指示にとどまり、「どの話題から掘るか」の手がかりが与えられていない。

キャラ側のシステムプロンプト（`SYSTEM_PROMPTS` の mint/claus/rain）には「会話の流れ」のステップが書かれているが、それはフェーズ1〜4を順番に辿る想定で、実際の会話で「今どのフェーズにいるか」をAIが自己判定する手がかりがない。結果として、モデルが保守的に広い質問へ戻る傾向がある。

### 原因分類
- **context**: 中盤以降の掘り下げの方向を示すコンテキストが薄い
- **prompt**: キャラのシステムプロンプト内「会話の流れ」がフェーズ番号列挙になっており、現在位置の自己判定を助ける構造になっていない

### 改善仮説
`buildInterviewQualityContext` の「中盤の進め方」に、すでに出ている話題から掘り下げ方向を絞る指示を追加する。

現状:
```
【中盤の進め方】
すでに出た話から、判断基準・こだわり・お客様の反応のどれかを1つだけ掘ってください。
```

改善案:
```
【中盤の進め方】
すでに出た話の中で、最も具体的だったエピソードを1つ選び、「なぜそうしたのか」「その判断を続けている理由」「相手がどう反応したか」のどれか1つを掘ってください。新しいエピソードを求めるより、出ているエピソードを深める方向を優先してください。
```

「最も具体的だったエピソードを1つ選び」という限定が、広い質問への揺り戻しを防ぐ。

### 変える場所
- ファイル: `lib/ai-quality.ts`
- 関数: `buildInterviewQualityContext`
- セクション: `input.userTurnCount >= 3` の条件分岐内のテキスト（165〜167行目）

### reviewに見てほしいこと
- 「新しいエピソードを求めるより出ているエピソードを深める」という方向転換の指示が、テーマが決まっていない「お任せ」インタビューで機能するか。テーマあり（suggested/custom）の場合と分けた方がよいか。
- 「最も具体的だったエピソード」という判断をモデルに委ねる形は適切か。「直近の会話で出たエピソード」に限定した方が安全か。

### 次回の確認ポイント
- 中盤（ターン3〜6）の質問が「エピソードを掘る」方向に絞られているか、それとも新しいエピソードを求める方向に戻っているか、実際の会話ログで確認する
- 改善前後で「中評価（もう一段深掘りできそう）」の割合が減るか、`interview-quality` ルーブリックで評価する

## AI教育ケース #2

### 症例
事業者が「丁寧にやっています」「お客様を大切にしています」など抽象語で答えたとき、キャラが「そうなんですね」と受け取り次の質問に進んでしまう（具体化できずに流す）ケースが設計上起きうる。

`lib/ai-quality.ts` の `looksAbstractResponse` 関数は抽象語を検出して「次は具体例を聞いてください」という指示をコンテキストに追加する仕組みがある。しかしこの指示は「インタビュアーが具体化しようとする」ことを促すのみで、「どういう質問の形で具体化するか」の型が示されていない。

結果として、モデルが「具体的に教えていただけますか？」「もう少し詳しく教えてもらえますか？」という汎用的な掘り下げに留まり、一次情報を引き出す力が弱くなる。

### 何が弱かったか
コンテキスト側の指示「次は『いつ・誰に・何をしたか』が出る具体例を1つだけ聞いてください」は方向性として正しい。しかし、キャラの人格（口調・フレームワーク）と対応していない汎用的な指示になっており、キャラごとの具体化の問い方が活かせていない。

ミントなら「最近、そう感じた場面が1つ出てきますか」、クラウスなら「その判断をした場面は、どういう状況でしたか」、レインなら「それを一番感じてもらえたお客様は、どんな方でしたか」という形で、キャラの口調で具体化の問いが出るのが理想だが、コンテキストの指示はキャラ非依存になっている。

### 原因分類
- **context**: 抽象回答への深掘り指示がキャラの人格と切り離されている
- **prompt**: キャラ別の「具体化の問い方の型」がシステムプロンプトに明示されていない

### 改善仮説
キャラのシステムプロンプト（`SYSTEM_PROMPTS`）の各キャラに「抽象語が出たときの問い直し方」セクションを追加する。コンテキスト側の指示はそのままにして、キャラ側に具体化の問い方の例を3つ持たせる。

ミントへの追加例:
```
【抽象語が出たときの問い直し方】
「丁寧」「安心」「大切に」などの言葉が出たら、次の3つのどれかで聞き直す。
- 「最近、それを感じてもらえた場面が1つ出てきますか？」
- 「そういう気持ちで動いたことが、最近でもありますか？」
- 「そのとき、お客様から何か言葉が返ってきましたか？」
```

1ケースにつき変更は1箇所。今回はミントのシステムプロンプトのみを変更対象とする。

### 変える場所
- ファイル: `lib/characters.ts`
- 対象: `SYSTEM_PROMPTS.mint` のプロンプト文字列
- 追加箇所: 「【絶対に言わないこと】」セクションの直前に「【抽象語が出たときの問い直し方】」を挿入

### reviewに見てほしいこと
- 「3つの問い直し例」を用意することで、モデルが毎回同じ例を選ぶ状態にならないか（バリエーションが消える問題）
- クラウス・レインにも同様のセクションを追加すべきか、それともミントで効果確認してから展開するか
- 既存の `CONVERSATION_QUALITY_INSTRUCTION`（`lib/characters.ts`）の「抽象的な答えには『最近の具体例』『そのときのやり取り』『誰に対してか』を聞く」との重複があるが、キャラ側に追加することで二重指示になり混乱しないか

### 次回の確認ポイント
- ミントのインタビューで「丁寧・安心・こだわり」などの抽象語が出た後、次の発言が「いつ・誰に・何をしたか」の具体を引き出す形になっているか確認する
- 改善後も「なるほど、丁寧なんですね」で流れてしまうケースが残るようなら、`CONVERSATION_QUALITY_INSTRUCTION` 側の強化を次ケースとして立てる

## AI教育ケース #3

### 症例
インタビューの充足判定（`SUFFICIENCY_INSTRUCTION`）が「7回返答した時点で条件確認」と「12回で強制終了」という回数ベースになっており、内容の質によらず終了提案が出る、または逆に良いタイミングで出ない、という状態が起きうる。

具体的には:
- 事業者が7回答えても全部が抽象的・短い場合（例: 「はい」「そうですね」の繰り返し）でも「十分な内容が集まりました」と誤判定される可能性がある
- 逆に5〜6回で非常に豊富な一次情報が出ているのに、7回に達するまで終了を提案できない

### 何が弱かったか
`SUFFICIENCY_INSTRUCTION`（`lib/characters.ts`）の充足判定条件は「事業者のこだわり・工夫・お客様への価値のいずれかが1点以上出ている」という質的な条件になっている。これは良い設計だが、その判定を行うのは回数トリガー（7回）がかかった時点のみで、それ以前は判定自体をしない。

一方で `buildInterviewQualityContext`（`lib/ai-quality.ts`）では「直近で聞けたこと」「次に聞くべきこと」を会話品質ガイドとして毎ターン注入している。この2つのレイヤーで「質の充足」と「回数のトリガー」が分断されており、整合が取れていない。

### 原因分類
- **structure**: 充足判定のトリガー（回数）と内容の質評価が別々のレイヤーに分離しており、連動していない
- **prompt**: `SUFFICIENCY_INSTRUCTION` の判定条件（「こだわり・工夫・価値が1点以上」）が曖昧で、モデルが「1点以上」を緩く判定しやすい

### 改善仮説
`SUFFICIENCY_INSTRUCTION` の充足条件の定義を具体化して、誤判定を減らす。

現状:
```
以下の情報が十分に集まっていれば、次のように終了を提案してください。
十分と判断する条件:
- 事業者のこだわり・工夫・お客様への価値のいずれかが1点以上出ている
```

改善案（条件に「具体性の要件」を追加）:
```
以下の情報が十分に集まっていれば、次のように終了を提案してください。
十分と判断する条件（すべて「具体的」であること）:
- 事業者のこだわり・工夫・お客様への価値のいずれかが1点以上出ている
- 「具体的」の基準: 実際の出来事・エピソード・言葉・行動・お客様からの反応のどれかが含まれている
- 抽象的な言葉のみ（「丁寧」「誠実」「こだわっている」）は1点と数えない
```

1ケースにつき変更は1箇所。変更対象は `SUFFICIENCY_INSTRUCTION` の定数のみ。

### 変える場所
- ファイル: `lib/characters.ts`
- 対象: `SUFFICIENCY_INSTRUCTION` 定数（199〜214行目）
- 変更箇所: 「十分と判断する条件」のテキスト部分に「具体的」の定義を追加する

### reviewに見てほしいこと
- 「抽象語は1点と数えない」という追加条件が強すぎて、事業者が誠実に答えているのに終了しない・会話が長引くという逆効果にならないか
- 「具体的」の定義例として挙げた「実際の出来事・エピソード・言葉・行動・お客様からの反応」はモデルが判定しやすい粒度か、それとも曖昧すぎるか
- `lib/ai-quality.ts` の `looksAbstractResponse` / `hasConcreteSignal` の判定ロジックと定義をそろえるべきか（現在は別々の語彙で「具体性」を定義している）

### 次回の確認ポイント
- 変更後、「こだわっています」「丁寧です」のみが7回続いたケースで終了提案が出なくなるか確認する
- 「抽象的な回答が多いのに終了を提案した」ケースが減るか、`interview-quality` ルーブリックの「低評価（一般論で終わっている）」件数で確認する

### ケース#1（lib/ai-quality.ts / buildInterviewQualityContext 中盤指示）

**判定: 条件付きGO**

**条件: 「最も具体的だったエピソード」を「直近の会話で出たエピソード」に変えること**

**根拠・検討内容**

改善仮説の方向性は正しい。「出ているエピソードを深める、新しいエピソードを求めない」という方向転換はinterview-qualityルーブリックの「中評価（もう一段深掘りできそう）」を減らす直接的な対処になっている。

ただし「最も具体的だったエピソードを1つ選び」という表現に問題がある。会話全体から「最も」を選ばせる判断はモデルに委ねる範囲が大きく、セッション序盤のエピソードを参照しようとして最近の文脈を見失うリスクがある。AIデザイナーが「直近の会話で出たエピソード」に限定した方が安全では、と自問しているが、その判断は正しい。`recentUserFacts`（直近3件）が既に`buildInterviewQualityContext`に注入されている構造と合わせると、「直近の会話で出たエピソード」に限定する方が既存設計と整合する。

「お任せ（omakase）」でも機能するかという問いについて。`focusTheme`がnullの場合でも`recentUserFacts`は機能しており、中盤指示は`focusTheme`と独立したブロックである。テーマなしでも動作上の問題はない。テーマあり・なしで中盤指示を分ける必要はない。

既存の`CONVERSATION_QUALITY_INSTRUCTION`との二重指示リスクは低い。`CONVERSATION_QUALITY_INSTRUCTION`は「具体例が出たら理由・判断基準・相手の反応のどれか1つを掘る」という一般原則であり、中盤指示は「エピソードの選び方」という絞り込みを追加する。役割が異なるため重複ではなく補完関係にある。希釈効果は限定的。

**変更条件の要約**
- 「最も具体的だったエピソードを1つ選び」→「直近の会話で出たエピソードを1つ選び」に変更すること

### ケース#2（lib/characters.ts / SYSTEM_PROMPTS.mint 抽象語問い直し追加）

**判定: 保留**

**保留理由: CONVERSATION_QUALITY_INSTRUCTIONとの実質的な二重指示になっており、追加の正当化根拠が弱い**

**根拠・検討内容**

`CONVERSATION_QUALITY_INSTRUCTION`の該当箇所を確認する。

```
- 抽象的な答えには「最近の具体例」「そのときのやり取り」「誰に対してか」を聞く
```

これはすでに「抽象回答への対処」として全キャラに適用されている共通指示である。今回ミントへ追加しようとしている「【抽象語が出たときの問い直し方】」は、この共通指示と重なる内容をキャラ固有の語彙で再定義するものになっている。

二重指示の問題は「混乱」ではなく「どちらを優先するかの曖昧さ」にある。共通指示が「最近の具体例・そのときのやり取り・誰に対してか」という3方向を示している状態で、ミント固有の3例を追加すると、モデルは合計6パターンから選ぶことになる。AIデザイナーが懸念している「毎回同じ例を選ぶバリエーション固定」のリスクは、選択肢が増えることでは解消しない。逆に増えることで選択の一貫性が崩れる可能性がある。

なぜキャラ側に追加するかの設計判断が弱い。「コンテキスト側の指示はキャラの人格と切り離されている」という分析は正しいが、今回の解決策（キャラ側に例を追加）がその問題を解決するかは別問題である。コンテキスト側の指示自体をキャラ人格に近づける方向（`buildInterviewQualityContext`内でキャラIDを受け取って指示文を分岐させる）の方が根本的であり、システムプロンプトを長くするリスクが低い。

ミントのシステムプロンプトはすでに長い（反応フレーズ10例＋会話の流れ4ステップ＋絶対言わないこと8項目＋共通4インストラクション）。さらにセクションを追加すると、既存の重要指示（「毎ターン受け取ってから質問」「1ターン3文以内」）が相対的に薄まる希釈効果が発生しうる。

**AIデザイナーへの返却内容**
- 共通の`CONVERSATION_QUALITY_INSTRUCTION`が既に「抽象回答への対処」を定義している。その指示が機能していないと判断した根拠（実際の会話ログ）が得られてから改善を設計すること
- キャラ側への追加より、`buildInterviewQualityContext`でキャラIDを受け取り`looksAbstractResponse`検出時の指示文をキャラ別に生成する方向を先に検討すること
- ミントで先行して実装し効果確認する方針は評価できる。ただし追加するセクション数と長さを最小化する設計を再提案してほしい

### ケース#3（lib/characters.ts / SUFFICIENCY_INSTRUCTION 充足条件具体化）

**判定: 条件付きGO**

**条件1: ファイナンスへのコスト影響確認を先に行うこと**
**条件2: 「具体的」の定義例の語彙を`hasConcreteSignal`のパターンと揃えること**

**根拠・検討内容**

症例の診断は正確である。現状の`SUFFICIENCY_INSTRUCTION`は「こだわり・工夫・価値が1点以上」という条件を持ちながら、モデルが「丁寧にやっています」を1点と数えることを防げていない。「抽象的な言葉のみは1点と数えない」という追加は、この誤判定を直接防ぐ設計になっている。

変更対象が`SUFFICIENCY_INSTRUCTION`の1箇所のみで、変更量が最小。他のインストラクションへの影響も限定的。設計の単純さは評価できる。

**「抽象語は1点と数えない」が会話を長期化させるリスクについて**

interview-qualityルーブリックの注意事項に「良いインタビュー = 長い会話ではない。必要な具体が取れていれば短くてもよい」とある。今回の変更は「具体が出るまで終了しない」という方向であり、この原則と整合する。ただし事業者によっては「ある程度抽象的な答えしか出ない」セッションが存在する。その場合に12回強制終了まで引き延びるのは事業者負担の最小化（CLAUDE.md判断原則1条）に反する可能性がある。

会話が7回から12回まで延びる確率が高まる場合、1セッションあたりのトークン消費が現状より増える。ターン延長が何パーセント発生するかは実走データがないため試算不可だが、「充足条件を厳しくする」変更であることからトークン増加方向に働くのは確実である。ファイナンスへの確認なしに進めるべきではない。

**`looksAbstractResponse` / `hasConcreteSignal`との定義不一致リスクについて**

`hasConcreteSignal`の判定パターン（`CONCRETE_SIGNAL_PATTERN`）は「数字・先月・先週・お客様・現場・施工・来店・そのとき・家族・具体」等の語彙ベースの正規表現である。`SUFFICIENCY_INSTRUCTION`の追加案に書かれた「実際の出来事・エピソード・言葉・行動・お客様からの反応」という定義例は、この正規表現が拾う語彙と対応していない。

「具体的」の基準が`lib/ai-quality.ts`（コード）と`SUFFICIENCY_INSTRUCTION`（LLMへの自然言語指示）で2系統存在することは設計上の問題ではないが、評価方向が食い違うと矛盾した動作を生む。改善案の定義例に「数字・日時・場所・お客様の言葉・具体的な状況の描写のいずれかが含まれている」など`hasConcreteSignal`の語彙に近い記述を使うことで、2つのレイヤーの基準を揃えることができる。

**変更条件の要約**
- ファイナンスへの確認: セッション長期化（7回→12回）によるトークンコスト増の影響試算を依頼し、問題なければ実装GOとする
- 「具体的」の定義例の語彙を`CONCRETE_SIGNAL_PATTERN`（ai-quality.ts）と整合させるよう定義文を修正すること

3件とも原因分類と変更箇所の特定は精度が高く、1ケース1変更の原則も守られている。ケース#1・#3は方向性が正しく条件付きで実装に進める。ケース#2はキャラ側への追加より先に取るべき手順がある。実際のインタビューログが蓄積されてから症例ベースに置き換えると、特にケース#3の充足条件の強度調整（「抽象語ゼロなら終了しない」が本当に正しいか）の検証精度が上がる。
