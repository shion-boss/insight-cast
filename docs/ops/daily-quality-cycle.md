# 日次品質改善サイクル

Phase 2 の毎日実施プロセス。9軸を網羅的にチェックし、漏れなく修正することでサービス全体を継続的に底上げする。

> **軸の設計方針（2026-04-25改訂）**
> - 軸1（UI）に表示速度チェックを統合（旧軸9を廃止）
> - 軸8（ドッグフーディング）をQA観点に絞り、進捗KPIは weekly-review.md へ移動
> - 軸9（新設）に非同期処理・通知品質を追加（メール・バックグラウンドジョブ）

## 目的

- 「使えるが荒い」状態から「使って気持ちいい」状態へ毎日1歩進める
- 問題を蓄積させない（週次で大掃除するより日次で小掃除する）
- ドッグフーディング取材を実際に回せる品質を維持する

---

## 9軸チェックリスト

### 軸1: UI（見た目・レイアウト・速度）
- [ ] 余白・フォント・色・アイコンに違和感がないか
- [ ] スマホ表示で崩れがないか（主要画面）、**修正後はPC側も必ず確認**
- [ ] ホバー・フォーカス・disabled 状態が適切か
- [ ] キャラアイコンが必要な場所に出ているか
- [ ] 画像に alt テキストがあるか（スクリーンリーダー対応）
- [ ] 体感で遅いページがないか（目安: 3秒以上は要調査）
- [ ] 静的コンテンツなのに `createClient()` + `getUser()` を呼んでいないか → CTAの出し分けだけならクライアントコンポーネントに移す
- [ ] 独立した複数のDBクエリが逐次実行になっていないか → `Promise.all` / `Promise.allSettled` で並列化
- [ ] 更新頻度が低いデータを毎回DBから取得していないか → `unstable_cache` + `revalidate` でキャッシュ化
- [ ] `Suspense` に fallback がなく JS 読み込み中が白画面になっていないか → スケルトンを設定する

### 軸2: UX（操作・導線・状態）
- [ ] 空状態・ローディング・エラーが無言になっていないか
- [ ] フォームの保存・未保存が分かるか
- [ ] 操作後に「次に何をすればいいか」が伝わるか
- [ ] URL パラメータが導線をまたいで引き継がれているか
- [ ] ログイン済み/未ログインで表示が正しく分岐しているか

### 軸3: 矛盾・整合性
- [ ] サービス説明・料金・機能の記述に矛盾がないか（CLAUDE.md が唯一の参照元）
- [ ] DB値とコード側の型・定数が一致しているか
- [ ] 公開ページ（LP/ブログ/特商法等）の情報が最新か
- [ ] キャラの役割・口調・設定に揺れがないか

### 軸4: AIキャストの品質
- [ ] インタビュー質問が浅くなっていないか（掘り下げているか）
- [ ] 最初のコメントが前置きなく質問に入っていないか
- [ ] キャラごとの専門軸（Industry Insight等）が出ているか
- [ ] プライバシー逸脱・スコープ外の発言がないか
- [ ] 記事生成の出力に違和感がないか

### 軸5: AI社員の品質
- [ ] エンジニアの実装に型安全な問題がないか
- [ ] AIデザイナーのプロンプトに意図しない副作用がないか
- [ ] コスト管理に異常値がないか（ファイナンス視点）
- [ ] CLAUDE.md・エージェントmdと実装が乖離していないか

### 軸6: コピー最適化
- [ ] 40〜60代AI未経験者に伝わる言葉になっているか
- [ ] 専門用語（プロンプト・生成・トークン・バッチ等）が顧客画面に出ていないか
- [ ] CTAラベルが「押すと何が起きるか」を一言で表しているか
- [ ] 世界観を壊す冷たい定型文がないか
- [ ] LP・ブログのコピーがメインキャッチコピーと整合しているか

### 軸7: セキュリティ
- [ ] APIルートに認証チェック（`auth.getUser()`）があるか
- [ ] `req.json()` から取る値に Zod バリデーションがあるか
- [ ] RLS が有効になっているテーブルで、意図しないデータ漏洩がないか
- [ ] XSS リスクのある `dangerouslySetInnerHTML` の使用箇所が安全か
- [ ] シークレット・APIキーがコードやログに出ていないか

### 軸8: 実使用フィードバック
実際にツールを使った時だけ記録する。進捗KPI（本数・セッション数）は weekly-review.md で管理。

- [ ] 今日ツールを使って気になった点・詰まった点があるか（`ops/dogfooding-log/` に記録）
- [ ] 40〜60代が使ったとして「分からない・迷う」場面がなかったか
- [ ] インタビュー中にAIキャストの応答が浅い・的外れと感じた箇所があったか
- [ ] 記事生成結果に手直しが必要な箇所があったか（その理由も記録する）

### 軸9: 非同期処理・通知品質
バックグラウンドジョブ・メール通知・Webhook など、画面外で起きる処理の品質確認。

- [ ] 競合調査・HP分析のバックグラウンドジョブが完了後にメール通知が届くか
- [ ] メール文面がキャラ名義になっているか（冷たいシステム文になっていないか）
- [ ] メール内のリンク（調査結果画面など）が正しく機能するか
- [ ] ジョブが失敗してもサイレントに落ちていないか（エラーログ・ユーザー通知があるか）
- [ ] Stripe Webhook の処理が冪等になっているか（二重実行で課金が重複しないか）
- [ ] バックグラウンド処理中に画面が「無言」になっていないか（ポーリング・進捗表示があるか）

## 起動責任者

**ディレクターがセッション開始時に起動する。人間が「チェックして」と言う前に回すのが原則。**

- エンジニアが UI 変更を含むコミットをした後は、その場でレビュアーに軸1（UI）の即時チェックを依頼する
- エンジニア自身も UI タスク完了時にモバイルチェック（`engineer.md` 参照）をセルフ実施し、結果を報告に含める
- 問題の指摘が人間から来た場合は、なぜチェックで検出できなかったかを振り返り、チェックリストを改善する

---

## 実施フロー

```
1. レビュアーに9軸チェックを依頼（主要画面・最近変更した箇所・最近使ったフローを中心に）
2. 検出した問題を優先度付け
   - Blocker（使えない・嘘の情報・セキュリティ穴）→ 当日修正
   - Should Fix（品質を下げている）→ 当日〜翌日
   - Nice to Have → 週次振り返りに積み上げ
3. エンジニア / AIデザイナー / マーケターに修正を依頼
4. 修正後、週次振り返り（ops/weekly-review.md）の「指摘パターン集計」に追記
```

---

## 記録ルール

- 当日修正したものは git commit メッセージに残す（`fix:` / `copy:` / `arch:` / `security:` 等）
- 繰り返し発生したパターンは CLAUDE.md またはエージェントmdにルール化を提案する
- 未解決・持ち越しは週次振り返りの「今週発生した課題・未解決事項」に記載する
- セッション終了前に必ずこのサイクルの実施結果を weekly-review.md に記録する
