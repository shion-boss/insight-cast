# 決定: アカウント削除フローの方針

## 日付

2026-04-29

## 背景

アカウント削除機能の実装にあたり、Stripe サブスクリプションの処理方針・エラー時の挙動・物理削除か論理削除か・削除確認UIの設計について方針が未確定だった。

レビュアーが実装上のBlocherとして以下を指摘し、人間が選択肢を判断・確定した。

- Stripe 解約タイミング: 即時 or 期末
- Stripe API エラー時の挙動: 削除を止めるか続行するか
- 削除方式: 物理削除か論理削除か
- 確認UIのデザイン: どこまでの情報を明示するか

## 検討した選択肢

### Stripe 解約タイミング

**選択肢A（採用）: 即時解約**
- 削除時点で Stripe のサブスクリプションをキャンセルし、それ以降の請求を止める
- ユーザーが意図的に削除を選んだのであれば即時が自然な挙動
- 期末まで利用権を残すと「削除したのにまだ使える状態」の混乱が生じる

**選択肢B: 期末解約**
- 解約月末まで利用権を残してから削除する
- 「払った分は使える」の期待に応えられるが、削除後もデータ・機能が残る状態が複雑

### Stripe エラー時の挙動

**5xx / ネットワークエラー時: 削除を止める（採用）**
- Stripe 側の一時障害や通信断では「解約が本当に通ったか」を確認できない
- 不明確な状態のまま削除を続行するとデータとStripe状態の不整合が生じる
- ユーザーに「しばらくしてから再試行してください」と案内する

**4xx エラー時: キャンセル済みとみなして削除続行（採用）**
- 4xx は「リクエストが届いた・処理できた（または処理不要）」を意味する
- already_canceled など「解約済み」を示す 4xx は削除を止める理由にならない
- そのまま削除フローを続行する

### 削除方式

**物理削除のまま（採用）**
- 論理削除（is_deleted フラグ等）への変更は実装コストが大きく、Phase 2 の方針に合わない
- 削除確認画面で「データが削除されること」「返金なしであること」を明示することでユーザーの誤操作リスクをカバーする

**論理削除への変更（不採用）**
- 誤削除の復元・監査に強いが、全テーブル・全クエリへの影響が大きく現時点では過剰

### 削除確認UI

**案A: 「データが消える」のみ明示**
- 削除のリスクだけ伝えるシンプルな構成

**案B（採用）: 「削除されること」「返金なしであること」を箇条書きで両方明示**
- 物理削除・即時解約・返金なしの3点をユーザーが確認してから実行できる
- 不可逆操作の確認として必要な情報が揃っている

## 決定内容

1. **Stripe 解約**: 削除時点で即時キャンセル
2. **Stripe 5xx / ネットワークエラー時**: 削除を中止し、ユーザーに再試行を案内する
3. **Stripe 4xx エラー時**: キャンセル済みとみなし、削除フローを続行する
4. **削除方式**: 物理削除のまま変更しない
5. **削除確認UI**: 案B — 「削除されること」「返金なしであること」を箇条書きで両方明示

## 理由

- 即時解約はユーザーの意図（削除＝利用を終える）と一致し、期末まで機能を残す複雑さを避けられる
- エラー種別でハンドリングを分けることで、不明な状態のまま削除を進めるリスクを排除しつつ、不要な中断も防ぐ
- 物理削除は現フェーズの実装コストと規模に対して適切。確認UIの情報明示でカバーする
- 確認画面の情報充足（削除・返金なしの明示）は CLAUDE.md ガードレール13条（ブラウザネイティブダイアログ禁止・不可逆操作は必ずインアプリUIで確認）とも整合する

## 影響を受ける場所

- アカウント削除 API（`/api/account/delete` 相当）のエラーハンドリング実装
- 削除確認モーダルのコピー・UIデザイン
- 将来 Phase 3 で課金フロー追加時: このログを参照して返金ポリシーとの整合を確認すること

## 見直し条件

- Stripe の仕様変更でエラーコード体系が変わった場合
- 論理削除が必要なユースケース（規約上の保持義務・監査要件等）が発生した場合
- 誤削除のサポート問い合わせが複数件発生した場合（論理削除の再検討トリガー）
