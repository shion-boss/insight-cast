# AI Education Playbook

## 目的

Insight Cast のプロダクトAIを、場当たり的な prompt 修正ではなく、再現可能な教育ループで育てる。

この文書でいう「教育担当」は `AIデザイナー` を指す。
`AIデザイナー` はプロダクトAIの設計責任者であると同時に、AIキャストと裏方AIの教育責任者でもある。

---

## 教育担当の役割

`AIデザイナー` の教育担当としての仕事は以下。

- 現場で起きた失敗や弱さを、教育課題として整理する
- 問題を `prompt` だけでなく、`context / rubric / structure / model / UI` のどこで直すべきか切り分ける
- 最小の変更単位で改善案を作る
- `レビュアー` と一緒に改善の妥当性を確認する
- 改善内容と残課題を次回に引き継げる形で記録する

---

## 教育データの入り口

教育材料は以下から集める。

1. `ops/feedback/ai-training-notes.md`
2. `docs/review-log/YYYY-MM-DD-<slug>.md`
3. 実際のインタビュー、要約、記事生成の出力
4. `マーケター` が見つけた反応や市場メモ

---

## 教育ループ

### 1. 症例収集

まず「うまくいかなかったケース」を集める。

例:
- 質問が浅く、一次情報が出なかった
- 回答を急かしてしまった
- 競合比較が一般論に寄った
- 記事化で会話にない事実を補ってしまった

### 2. 失敗の分類

原因を以下のどれかに切る。

- `prompt`: 指示が足りない、強すぎる、曖昧
- `context`: 必要な前提情報が渡っていない
- `rubric`: 良し悪しの基準が弱い
- `structure`: 入出力の形が悪い
- `model`: 必要な知能や安定性に足りない
- `ui`: ユーザー入力や見せ方が原因

### 3. 改善案の設計

改善は一度に盛りすぎない。

- 変更は 1症例につき 1仮説を基本にする
- prompt と rubric を同時に変える場合も、狙いを分けて書く
- 再現確認できない大改修を避ける

### 4. レビュー依頼

改善案は `レビュアー` に渡して以下を確認する。

- 症例に効いているか
- 他の品質を壊さないか
- 世界観やユーザー像から外れていないか

### 5. 記録

以下を残す。

- どんな症例を見たか
- 何を原因と見たか
- 何を変えたか
- 何がまだ残っているか

---

## 直し方の原則

### prompt だけで直そうとしない

弱い出力を全部 prompt の責任にすると、設計が肥大化する。
必要なら `エンジニア` に context や UI の変更を依頼する。

### 症例ベースで育てる

「もっと賢くする」ではなく、
「この失敗を次に起こしにくくする」
で育てる。

### 教育と評価を分ける

`AIデザイナー` は育てる側。
`レビュアー` は通すか止めるかを判定する側。
同じ AI が自分で育てて自分で合格にしない。

### 教育の単位を小さくする

一度に複数キャラを大きく変えない。
まず 1キャラ、1機能、1失敗パターンで改善する。

---

## 記録先

- 現場メモ: `ops/feedback/ai-training-notes.md`
- 設計変更: `docs/ai-specs/<feature>.md`
- レビュー結果: `docs/review-log/YYYY-MM-DD-<slug>.md`
- 運用整理: `docs/ai-team/operating-model.md`

---

## よくある教育課題

### インタビューが浅い
- 抽象語の具体化が足りない
- 深掘りの順番が弱い
- 会話相手の語彙を拾えていない

### 分析が一般論に寄る
- 対象サイト固有の根拠を拾えていない
- 業界常識と対象差分の切り分けが弱い

### 記事が盛られる
- 事実境界の指示が弱い
- 出力構造が自由すぎる

### キャラの差が薄い
- 共通原則が強すぎる
- キャラ固有の得意領域が prompt に反映されていない

---

## 教育担当の完了条件

教育担当としての改善は、以下を満たした時に一段落とみなす。

- 症例と原因が記録されている
- 改善案が最小単位で言語化されている
- `レビュアー` の確認が入っている
- 次に見るべき症例が明確
