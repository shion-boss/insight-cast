# AI Team Operating Model

## 目的

Insight Cast の社内AI社員が、役割の重複や引き継ぎ漏れを起こさずに動ける状態を作る。

この文書は以下を揃えるために使う。

- だれが何に強いか
- どの仕事で誰が主担当か
- どう連携すれば詰まりにくいか
- どの shared skill を使えば品質を揃えられるか

## 担当の基本線

### `ディレクター`
- 方針整理、優先順位、仕様起票、決定ログ、週次レビューの司令塔
- 他エージェントに仕事を振る時の入り口になる

### `エンジニア`
- 画面、API、DB、認証、インフラなどプロダクトの器を実装する
- AI機能では `AIデザイナー` とセットで動く

### `AIデザイナー`
- プロダクトAIの知性を設計する
- 質問設計、分析観点、モデル案、評価基準の初版を作る
- AI教育担当として、失敗症例から改善ループを回す

### `レビュアー`
- コード品質、AI出力品質、世界観の一貫性を止める役
- 通すか戻すかの判定線を持つ

### `マーケター`
- 自社HP、コピー、獲得施策、中間マイルストーンの追跡を担う
- 成果の見立てを `ディレクター` に返す

### `オペレーター`
- 障害、問い合わせ、ランブック、FAQ の運用を担う
- バグを直すのではなく、再発しにくい流れを作る

### `finance`
- 固定費モード、モデルコスト、月次コストレポートを担う
- `AIデザイナー` のモデル案に対する歯止め役になる

## 連携フロー

### 新機能
1. `ディレクター` が依頼を要約し、目的と完了条件を揃える
2. 仕様が要る場合は `spec-template` で骨子を作る
3. AI機能なら `エンジニア` と `AIデザイナー` が先に入出力を合わせる
4. 実装後は `エンジニア` / `AIデザイナー` から `レビュアー` へ handoff する
5. 仕様や方針が変わったら `ディレクター` が決定ログを更新する

### インシデント
1. `オペレーター` が `incident-triage` で優先度と影響範囲を切る
2. 修正が要れば `エンジニア` へ handoff する
3. AI挙動の問題なら `AIデザイナー` へ切る
4. 収束前に `レビュアー` が再発観点を確認する
5. `オペレーター` が記録と FAQ / runbook 反映を行う

### モデル変更
1. `AIデザイナー` が品質要件と候補モデルを整理する
2. `finance` が `model-cost-review` でコスト影響を算定する
3. `ディレクター` が優先順位と導入タイミングを整理する
4. 人間が最終承認する

### AI教育
1. `オペレーター` と `レビュアー` が症例を `ops/feedback/ai-training-notes.md` と `docs/review-log/` に残す
2. `AIデザイナー` が `ai-education-loop` で原因分類と改善仮説を作る
3. `レビュアー` が改善案の妥当性を確認する
4. `ディレクター` が必要なら優先順位と影響範囲を整理する

### 自社グロース
1. `マーケター` が中間マイルストーン進捗を更新する
2. `finance` がコスト変化を確認する
3. `ディレクター` が週次レビューに統合する

## 運用リズム

### 毎回
- 依頼の要約
- 完了条件の確認
- handoff 時の前提、未解決、確認済み事項の明記

### 週次
- `ディレクター`: `ops/weekly-review.md` 更新
- `マーケター`: 中間マイルストーンの進捗更新
- `オペレーター`: インシデントと FAQ の差分確認

### 月次
- `finance`: `docs/cost-report/YYYY-MM.md` 更新
- `ディレクター`: 重要な決定と実績のズレを確認

## 今回のチェック結果

### 強い点
- 役割分担そのものはかなり明確
- `エンジニア` と `AIデザイナー` の境界線は言語化されている
- `レビュアー` が世界観まで止める構造になっている

### 弱かった点
- handoff の型が agent ごとに散っていた
- 決定ログ、週次レビュー、マイルストーン管理が個人技になりやすかった
- `オペレーター` と `finance` に shared skill がなく、再利用しにくかった
- 既存知識の一部が `README.md` 止まりで、skill として使いにくかった

### 補填方針
- 共通運用 skill を追加して、連携の型を揃える
- 既存知識に `SKILL.md` を足して発火しやすくする
- agent 定義から shared skill を明示して迷いを減らす
