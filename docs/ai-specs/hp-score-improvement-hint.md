# HP評価スコア 改善条件設計

設計担当: AIデザイナー
作成日: 2026-05-01
対象機能: `/projects/[id]/report` のHP評価スコア表示

---

## 目的

`hp_audits.raw_data.site_evaluation` の各スコアについて、
「今のスコアが出た根拠」と「次のスコア帯に上がるために必要な行動」を
ユーザーが理解できる形で提示する。

ユーザーが「4から5にするには何が必要か」を見て、
毎月の改善を楽しみながらHPを育てられる体験を作る。

---

## 設計の背景

### Insight Cast が提供できる成果物の範囲

Insight Cast がユーザーに届けられる成果物は「インタビューで引き出した一次情報をもとに書くブログ記事」のみ。
HP本体のトップページ改修・実績ページ整備・CTA設計などは提供対象外。

そのため `improvement_hint` が「トップページの文言を直してください」「実績ページを作ってください」になってはいけない。

### スコアの起点と上限

- スコアの起点はサイト全体（非ブログページ + 既存ブログ）で決まる
- HP本体が充実しているサイトは初回スコアが高く出る。これは正しい評価
- 非ブログ要因によってスコアが7〜8になることは許容する
- スコアが高いこと自体は問題ではない。問題は「hint がHP改修を促すこと」

### `improvement_hint` のスコア帯別優先順位

| スコア帯 | improvement_hint の方向 |
|---|---|
| 1-6 | 必ずブログ・インタビュー起点の提案にする |
| 7-8 | ブログ提案 **または** HP本体への小さな改善提案のどちらでもよい |
| 9-10 | 「この調子で続けましょう」「現在の水準を維持してください」でよい |

スコア1〜6でHP本体の改修を提案してしまうと、Insight Cast が提供できない行動を勧めることになり、ユーザーが「で、Insight Cast は何をしてくれるの?」と感じてしまう。

## 対象の評価軸

現行プロンプトが出力する6軸:

| key | label |
|---|---|
| `positioning` | ポジショニングの明確さ |
| `offer` | 提供価値の伝わりやすさ |
| `trust` | 信頼材料の厚み |
| `cta` | 問い合わせ導線 |
| `blog_coverage` | ブログの網羅性 |
| `blog_depth` | ブログの一次情報度 |

## 出力フォーマットの変更

### 変更前（現行）

```json
{
  "key": "blog_coverage",
  "label": "ブログの網羅性",
  "score": 4,
  "summary": "月1本未満の散発的な更新が見られます。"
}
```

### 変更後（設計案）

```json
{
  "key": "blog_coverage",
  "label": "ブログの網羅性",
  "score": 4,
  "summary": "月1本未満の散発的な更新が見られます。",
  "improvement_hint": "月1本以上のペースで3ヶ月続けると、このスコアが上がります。"
}
```

### `improvement_hint` の要件

- 1文で具体的に書く（「コンテンツを充実させる」のような曖昧な表現はNG）
- ユーザーが「次の取材・記事で実行できるアクション」として読める内容にする
- スコアが10のときは「現在の水準を維持してください」と固定する
- 40-60代のIT未経験者に伝わる言葉で書く（専門用語・横文字を使わない）
- 「改善してください」のような命令口調にしない。「〜すると上がります」の形を基本とする
- スコア1〜6は必ずブログ・インタビュー起点の提案にする
- スコア7〜8はブログ提案またはHP本体への小さな改善提案のどちらでもよい

### 禁止パターン

以下の文言はいかなるスコア帯でも出力しない:

- HP構造の変更・全面リニューアルを促す文言（「HPを作り直す」「サイト全体を見直す」「HP設計を変更する」等）
- HP制作会社への依頼を促す文言（「専門家に相談する」「制作会社に依頼する」「プロに頼む」等）
- スコア1〜6でのHP本体改修提案（「トップページに〜を加える」「実績ページを作る」「CTAボタンを置く」等）

## 評価軸ごとのスコアルーブリック

スコアは1〜10の整数。スコア帯で定義する。

---

### 1. ポジショニングの明確さ（`positioning`）

「このHPを初めて見た人が、何をしている会社でどんな人向けかを30秒で把握できるか」を評価する。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | 業種や提供内容がほぼ分からない。会社名と連絡先しかない。 | ブログ起点: 「誰向けの仕事が多いか、取材で話してブログにまとめましょう」 |
| 4-6 | 業種は分かるが「どんな人向けか」「他社との違い」が不明。もしくは伝わるが「なぜこの会社に頼むのか」が薄い。 | ブログ起点: 「なぜその仕事を選んでいるかを取材で聞き出してブログにすると伝わります」 |
| 7-8 | ターゲット・提供内容・差別化の3点が概ね伝わる。 | ブログ提案 または: 「トップページに一言ターゲットを書くとより伝わります」 |
| 9-10 | 30秒で全てが分かる。競合との差も明確。 | 「この調子でブログを続けましょう」 |

**ブログ経由スコアアップパス（1〜6帯）:**
- 誰向けの仕事が多いかを語った記事 → 読者に「自分向けのサービスだ」と伝わる
- 仕事を選んだ理由・経緯を語った記事 → 「なぜこの会社に頼むのか」の補完になる

---

### 2. 提供価値の伝わりやすさ（`offer`）

「このサービスを使うと自分の生活・仕事がどう変わるか」がHPから読み取れるかを評価する。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | 機能・メニューの列挙のみ。使った後どうなるかの記述がない。 | ブログ起点: 「お客様がどう変わったかを取材してブログにまとめましょう」 |
| 4-6 | 効果の記述はあるが漠然としている、または根拠（事例・数字）が薄い。 | ブログ起点: 「最近のお客様に起きた変化をインタビューして記事にすると伝わります」 |
| 7-8 | 具体的な効果と根拠が概ね揃っている。 | ブログ提案 または: 「サービスページに利用後の変化を一行添えるとスコアが上がります」 |
| 9-10 | 価値・根拠・共感の3点が揃い、初見で申し込みたくなる。 | 「この調子でブログを続けましょう」 |

**ブログ経由スコアアップパス（1〜6帯）:**
- お客様の変化・成果を語った記事 → 「使うとどうなるか」を補完する
- 「依頼前と依頼後」を比較した記事 → 提供価値の具体化につながる

---

### 3. 信頼材料の厚み（`trust`）

「初めて見た人が「ここに頼んでいいのか」と判断できる材料があるか」を評価する。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | 実績・事例・口コミ・資格など信頼につながる情報が何もない。 | ブログ起点: 「お客様の声をインタビューして1本まとめましょう」 |
| 4-6 | 創業年・所在地など最低限の情報はあるが実績が見えない。または実績・口コミはあるが古いか量が少ない。 | ブログ起点: 「事例や資格・経験を語った記事を取材で引き出しましょう」 |
| 7-8 | 実績・事例・口コミが概ね揃い、信頼感が高い。 | ブログ提案 または: 「実績ページに事例を1件追加するとスコアが上がります」 |
| 9-10 | 数字・事例・顔・口コミが揃い、初見で安心して問い合わせできる。 | 「この調子でブログを続けましょう」 |

**ブログ経由スコアアップパス（1〜6帯）:**
- お客様の声・体験談を取材した記事 → 口コミの補完になる
- 資格・実績・経歴を語った記事 → 専門性・信頼の根拠になる
- 「大変だった仕事とその解決」の記事 → 実力を示すエピソードになる

---

### 4. 問い合わせ導線（`cta`）

「読んだ人が次のアクション（問い合わせ・予約・電話など）に自然に進めるか」を評価する。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | 問い合わせ先が見つからない、または見つけにくい。 | ブログ起点: 「記事の最後に『気になった方はこちらから』と一言添えましょう」 |
| 4-6 | 問い合わせ先はあるがページ下部のみで気づきにくい。または配置はあるが文言の動機づけが弱い。 | ブログ起点: 「ブログ末尾に連絡先への案内を入れると導線が生まれます」 |
| 7-8 | 複数箇所にCTAがあり、文言も具体的。 | ブログ提案 または: 「トップページにわかりやすい問い合わせボタンを1つ置くとスコアが上がります」 |
| 9-10 | 読み進めるたびに自然に問い合わせしたくなる設計。ハードルが低い。 | 「この調子でブログを続けましょう」 |

**ブログ経由スコアアップパス（1〜6帯）:**
- 記事末尾への連絡先・問い合わせ案内の追記 → ブログが導線を増やすページになる
- 「よくある相談と答え」の記事 → 問い合わせへの心理的ハードルを下げる

---

### 5. ブログの網羅性（`blog_coverage`）

「サービスに関心がある人が知りたいテーマをブログでカバーできているか」を評価する。
更新頻度・テーマの幅の両方を見る。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | ブログがない、または年0-2本しかなく実質停止している。 | まず1本書くことが出発点。今日一番多く聞かれることをテーマに選ぶとよい |
| 4-5 | 月1本未満の散発的な更新。テーマが偏っていたり古い記事のまま止まっている。 | 月1本のペースで3ヶ月続けると、このスコアが上がる |
| 6-7 | 月1-2本の安定更新はできているが、テーマが似た内容に偏っている。 | よくある質問・サービスの使い方・事例の3種類のどれかを1本追加するとよい |
| 8-9 | 月2-3本更新でき、複数のテーマを扱っている。一次情報を含む記事がある。 | 取材記事か実例紹介をあと1本追加するとスコアが10に近づく |
| 10 | 月3本以上、インタビュー記事や事例も含む継続更新が実現している。 | 現在の水準を維持してください |

---

### 6. ブログの一次情報度（`blog_depth`）

「自社・自分にしか書けない情報がブログに入っているか」を評価する。
一般的な解説や情報まとめだけでなく、実体験・現場の声・固有の判断基準が含まれるかを見る。

| スコア帯 | 状態 | `improvement_hint` の方向 |
|---|---|---|
| 1-3 | どこでも読める一般情報・まとめ記事のみ。自社固有の情報が見当たらない。 | 「最近お客様に言われたこと」か「現場で気をつけていること」を1つ書いてみるとよい |
| 4-5 | 自社の名前は入っているが内容は一般論で、読んでも他社との違いが伝わらない。 | 「なぜ自社はそのやり方をしているか」の理由を1つ加えると深みが出る |
| 6-7 | 実体験や工夫の記述はあるが、読者が「この会社ならでは」と感じる部分が薄い。 | お客様とのやりとりや現場のエピソードを1つ盛り込むとスコアが上がる |
| 8-9 | 自社の経験・事例・判断基準が入っている。読んで「なるほど、この会社に頼みたい」と思える記事がある。 | 代表者や担当者の言葉を引用した取材記事を1本追加するとより強くなる |
| 10 | 自社にしか書けない情報が複数の記事で継続的に発信できている。 | 現在の水準を維持してください |

## モデル選定

現行: `claude-sonnet-4-6`（analyzeHp 関数内）

`improvement_hint` の追加は出力フィールドが1つ増えるのみで、
タスクの性質・必要な知能レベルは変わらない。モデル変更は不要。

ただし `max_tokens: 3000` は `improvement_hint` 6フィールド分（1フィールドあたり約50トークン想定 x 6 = +300トークン）で上限に当たる可能性がある。
エンジニアに `max_tokens: 3500` への引き上げを依頼すること。

最終承認は人間（コスト絡み）。ファイナンスにコスト影響の確認を依頼すること。
