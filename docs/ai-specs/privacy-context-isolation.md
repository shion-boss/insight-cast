## PRIVACY_SCOPE_INSTRUCTION が守るリスクの設計意図

`PRIVACY_SCOPE_INSTRUCTION` は2種類の異なるリスクを同時に防ぐために存在する。

### リスク1: マルチテナント混在（実装ではなくモデル行動の問題）

DB・APIレイヤーでは `project_id` フィルターにより A 社のデータが B 社のセッションに渡ることはない。しかし LLM は「コンテキストに含まれる情報」を「業界全般に通じる知識」と誤解することがある。たとえばクラウスがある整骨院の競合調査結果を読み込んだ後、その情報を「整骨院ではよくあること」として語り始めるケースがこれに当たる。

これは「情報が漏洩している」のではなく「モデルが情報の帰属を見失っている」状態であり、ユーザー（事業者）から見ると「なぜその情報を知っているのか」という不安・不信感につながる。

### リスク2: 一般論の流用防止（引き出しの質の問題）

競合調査結果や HP 分析が「業界の一般論」として語られると、インタビューが「答え合わせ」になる。事業者本人の中にしかない一次情報を引き出すという目的から外れ、調査結果の裏付け取りになってしまう。

このリスクは `[H] 調査結果の活用バランス` 評価軸とも対応している。プロンプトレベルで「このセッションで得た情報を根拠にしない」と明示することで、モデルが調査結果を普遍的事実として扱う行動を抑制する。

### まとめ

| 防ぐリスク | 発生源 | 影響 |
|---|---|---|
| マルチテナント混在 | モデルがコンテキスト情報の帰属を見失う | ユーザーの不信感・プライバシー懸念 |
| 一般論の流用 | モデルが調査結果を普遍的事実として語る | インタビューの質低下・答え合わせ化 |

両リスクとも「実装バグ」ではなく「モデルの推論特性」に起因する。プロンプトで明示的に抑制するのが現時点の最善策。

---

## プロンプト文面の問題（要対応）

### 現状の不足点

1. SYSTEM_PROMPTS のどのキャラにも「他の取材先の情報を参照しない」旨の明示がない
2. コンテキスト注入ブロックに「これは今日の取材先専用の情報です」という帰属が明記されていない
3. CONVERSATION_QUALITY_INSTRUCTION と SUFFICIENCY_INSTRUCTION は存在するが、情報スコープを示すインストラクションが欠けている

### リスクの性質

- モデルが調査結果ブロックの情報を「業界全般の話」として誤用することがある
- 「他の会社でも〜」「一般的に〜」という形で、コンテキスト由来の情報を普遍的な事実として語るターンが起こりうる
- ユーザー（取材先事業者）から見ると「なぜそれを知っているのか」という不安につながる可能性がある

---

## 修正案

### A. 共通インストラクション定数の追加（`lib/characters.ts`）

`CONVERSATION_QUALITY_INSTRUCTION` と同じ方式で切り出す。

```typescript
const PRIVACY_SCOPE_INSTRUCTION = `

【情報のスコープについて】
- このセッションで共有された情報は、今日の取材先のものだけです
- 他の取材先・他社の事例として語らない
- 「他の会社でも〜」「一般的に〜」と言う場合でも、このセッションで得た情報を根拠にしない
- コンテキストに含まれる調査結果・競合情報は今日の取材先専用の情報です`
```

各キャラの SYSTEM_PROMPTS 末尾に `+ PRIVACY_SCOPE_INSTRUCTION` を追加する。

### B. コンテキスト注入ブロックへの帰属明示（`chat/route.ts`）

`contextParts` の冒頭に以下を追加する。

```typescript
if (projectData) {
  contextParts.push(
    `【このインタビューのスコープ】\n以下の情報はすべて今日の取材先「${projectData.name ?? '未設定'}」のものです。他社・他の取材先の情報は含まれていません。このセッションの外の情報には言及しないでください。`
  )
}
```
