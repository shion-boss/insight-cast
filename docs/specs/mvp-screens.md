# Insight Cast MVP 画面仕様書

## 作成日
2026-04-17

## ステータス
確定（実装着手可能）

## 前提条件
- Next.js 15 (App Router) + TypeScript strict + Tailwind CSS + Supabase
- 最初から顧客が使う前提のプロダクト（内部ツールではない）
- 自分たち自身も同じツールを使って自社ブログを更新する
- 無料インタビュアー: ミント（猫）/ クラウス（フクロウ）/ レイン（キツネ）
- 会話ブロックの左右: 左=回答者（ユーザー）、右=インタビュアー（キャラ）

### 画面1: ランディングページ `/`

**役割**
サービスの価値を伝え、登録・ログインに誘導する。AIツール感を出さない。「AIキャストが話を聞いてくれる会社」として見せる。

**UI要素**

```
[ヘッダー]
  - ロゴ（Insight Cast）
  - ログインリンク
  - 「試してみる」ボタン（CTAボタン）

[ヒーローセクション]
  - メインコピー: 「あなたにとっての当たり前は、まだ伝わっていない価値かもしれません。」
  - サブコピー: 「Insight Cast は、AIインタビューでそれを引き出し、ホームページを少しずつ強くします。」
  - CTAボタン: 「まずは無料で試してみる」
  - キャラ集合イラスト（assets/characters/portraits/）

[サービス説明セクション]
  - 3ステップ説明（キャラアイコン付き）:
    Step1: [クラウスアイコン] あなたのHPと競合を調査します
    Step2: [ミントアイコン] インタビュアーが取材に来ます
    Step3: [レインアイコン] 記事になる素材を整理します

[インタビュアー紹介セクション]
  - ミント・クラウス・レインの3カード
  - 各キャラ: イラスト + 名前 + 一言紹介 + 得意なこと

[記事サンプルセクション]
  - 通常記事サンプル
  - インタビュー風記事サンプル（会話ブロック含む）

[フッター]
  - ログインリンク
  - 利用規約
  - プライバシーポリシー
```

**設計注意点**
- 「AI」「生成」「プロンプト」の語を出さない
- キャラアイコンが世界観を担う。テキストは分かりやすさ優先
- 40-60代が「自分のことだ」と3秒で感じるコピー

### 画面2-3: 認証画面 `/auth/signup` `/auth/login`

**役割**
Supabase Auth を使ったシンプルな認証。

**UI要素（登録）**
```
- メールアドレス入力
- パスワード入力（8文字以上）
- パスワード確認入力
- 「登録する」ボタン
- 「すでにアカウントをお持ちの方はこちら」リンク
```

**UI要素（ログイン）**
```
- メールアドレス入力
- パスワード入力
- 「ログインする」ボタン
- 「パスワードを忘れた方はこちら」リンク
- 「新規登録はこちら」リンク
```

**設計注意点**
- エラー文は具体的に: 「メールアドレスまたはパスワードが正しくありません」
- 「アカウント作成」ではなく「登録する」
- ログイン成功後は `/dashboard` にリダイレクト

### 画面4: ダッシュボード `/dashboard`

**役割**
ログイン後の起点。プロジェクト一覧と新規作成への導線。「ダッシュボード」らしく見せない。

**UI要素**
```
[ヘッダー]
  - ロゴ
  - アカウントメニュー（ログアウト・設定）

[メインエリア]
  - 「新しいプロジェクトを始める」ボタン（大・目立つ）
  - プロジェクトカード一覧:
    - プロジェクト名
    - 自社HP URL
    - 最終更新日
    - ステータスバッジ:
        「調査中」（背景処理実行中）
        「インタビュー待ち」（レポート確認済み）
        「インタビュー済み」
        「記事生成済み」
    - 「続ける」ボタン

[空状態（プロジェクト0件）]
  - ミントのアイコン
  - 「まだプロジェクトがありません。最初の取材を始めてみましょう。」
  - 「プロジェクトを始める」ボタン
```

**ステータス遷移**
```
作成直後 → 「調査中」
調査完了 → 「インタビュー待ち」
インタビュー完了 → 「インタビュー済み」
記事生成完了 → 「記事生成済み」
```

**設計注意点**
- グラフや数値の羅列は出さない
- 「ダッシュボード」という言葉・UIにしない
- カードは最大4列程度でシンプルに

### 画面5: プロジェクト作成 `/projects/new`

**役割**
自社HP URLの登録。調査開始のトリガー。

**UI要素**
```
[クラウスのアイコン + ガイドメッセージ]
  「調査したいホームページのURLを教えてください。」

[入力フォーム]
  - プロジェクト名（任意・例: 「〇〇塗装店のHP」）
  - 自社HP URL（必須）
  - 「次へ進む」ボタン

[補足テキスト]
  「URLを登録すると、クラウスがホームページの調査を始めます。
   競合サイトは次のステップで登録できます。」
```

**バリデーション**
- URL は https:// がなくても補完する
- 到達不能な URL はエラー表示（ただしMVPでは簡易チェックのみ）

**処理**
- DB に project を INSERT
- 次の画面（競合URL登録）に遷移
- HP調査の非同期ジョブを開始（後ほど完了通知）

### 画面6: 競合URL登録 `/projects/[id]/competitors`

**役割**
比較対象を最大3件登録。スキップ可能。

**UI要素**
```
[クラウスのアイコン + ガイドメッセージ]
  「同業他社のホームページがあれば教えてください。
   比較することで、あなたのHPに何が足りないかが見えてきます。」

[入力フォーム]
  - URL入力欄（1件目、必須ではない）
  - 「+ もう1件追加する」リンク（最大3件）
  - 「調査を始める」ボタン
  - 「競合が分からないのでスキップする」リンク

[補足テキスト]
  「分からなければスキップしても大丈夫です。
   あなたのHPの調査だけ行います。」
```

**処理**
- DB に competitors を INSERT
- 「調査を始める」で調査・比較レポート画面に遷移
- 競合調査の非同期ジョブを開始（HP調査と並行）

### 画面7: 調査・比較レポート `/projects/[id]/report`

**役割**
自社HP分析と競合比較の結果を表示。「何が足りないか」を伝える。

**UI要素 - 調査中状態**
```
[クラウスのアイコン（アニメーション）]
  「クラウスが調査しています。
   完了したらメールでお知らせします。
   このまま待つこともできます。」

[プログレス表示]
  - 「自社HPを読み込んでいます...」
  - 「競合サイトを比較しています...」
  など段階的なステータス表示

[ダッシュボードに戻るリンク]
```

**UI要素 - 調査完了状態**
```
[クラウスのアイコン]
  「調査が完了しました。結果をご覧ください。」

[セクション1: あなたのHPの現状]
  - 現在伝えていること（箇条書き 3点程度）
  - 強みとして見えるもの（箇条書き 2-3点）
  - 伝えきれていないこと（箇条書き 2-3点）

[セクション2: 競合との比較]（競合登録した場合のみ）
  - 競合が伝えていてあなたのHPにないこと（箇条書き）
  - あなたのHPが競合より詳しいこと（箇条書き）

[セクション3: インタビューで深めたいテーマ]
  - 提案テーマ 3〜5個（クリックして詳細展開も可）

[「インタビューを始める」ボタン（大）]
[「URL を修正する」リンク]
```

**設計注意点**
- テーブルやグラフは使わない。文章と箇条書き
- 「スコア」「点数」は出さない
- 大事なことだけ。長くなりすぎない
- キャラアイコンが各セクションに存在感を持つ

**データ**
- `hp_audits` テーブル: 自社HP分析結果を保存
- `competitor_analyses` テーブル: 競合比較結果を保存

### 画面8: インタビュアー選択 `/projects/[id]/interviewer`

**役割**
3体のキャラから1体選ぶ。「キャストを選ぶ」体験の核。

**UI要素**
```
[見出し]
  「今日はどのインタビュアーに来てもらいますか?」

[キャラカード × 3]
  各カード:
    - キャラの全身イラスト（portraits/）
    - 名前（ミント / クラウス / レイン）
    - 一言紹介
    - 得意なこと（引き出すもの）
    - 選択ボタン「来てもらう」

  ミント（猫）:
    「親しみやすい雰囲気で、お客様目線の話を引き出します」
    得意: 安心感・気づかい・サービスの温かさ

  クラウス（フクロウ）:
    「業種の知識をもとに、技術的な違いを掘り起こします」
    得意: 専門性・判断基準・他社との違い

  レイン（キツネ）:
    「マーケティング視点で、差別化ポイントを引き出します」
    得意: 訴求ポイント・強みの言語化・差別化

[将来追加予定の予告（薄く表示）]
  「ハル・モグロ・コッコ: coming soon」

[「このインタビュアーに来てもらう」ボタン]
  （選択したカードがハイライトされたら有効化）
```

**設計注意点**
- 「選択する」ではなく「来てもらう」
- スペックの羅列にしない。得意なことが1行で伝わればOK

### 画面9: インタビュー `/projects/[id]/interview`

**役割**
AIとの対話。Insight Cast の中核体験。LINEのようなチャット UI。

**UI要素**
```
[ヘッダー（固定）]
  - 選択したキャラのアイコン + キャラ名
  - 「インタビューを終わらせる」ボタン（右上・目立たせすぎない）

[対話エリア（スクロール可能）]
  - キャラの発言: 右寄り吹き出し + キャラアイコン
  - ユーザーの発言: 左寄りシンプル吹き出し
  - ローディング: キャラアイコン + 「...」アニメーション

[進捗表示（固定・上部）]
  固定問数ではなく状態ベース:
    「話を聞かせてもらっています」→
    「もう少し教えてもらえますか」→
    「いい話が集まってきました」→
    「まとめに入ります」

[テキスト入力エリア（固定・下部）]
  - テキスト入力欄（複数行可）
  - 送信ボタン
  - Enterで送信（Shift+Enterで改行）

[モグロ選択時の例外]
  - テキスト入力欄は表示しない
  - 「はい」「いいえ」の固定ボタンで回答する
  - 回答後、モグロが「ここから分かること」を短く言語化して次の質問へ進む
```

**インタビュー終了の判断**
- 「充分な回答が得られたか」をAIが判断（固定問数ではない）
- AIが「整理に入る」と判断したら、ユーザーに確認:
  「いいお話がたくさん聞けました。これを記事の素材にまとめてもいいですか?」
  → 「はい、まとめてください」ボタン
  → 「もう少し話す」ボタン
- ユーザーが途中で「インタビューを終わらせる」を押した場合も同じ確認を挟む

**データ**
- `interviews` テーブル: セッション単位
- `interview_messages` テーブル: Q&A を1件ずつ保存

### 画面10: インタビュー整理結果 `/projects/[id]/summary`

**役割**
インタビューで引き出した内容を整理したテキストを確認。記事生成前の「素材確認」。価値を感じさせる見せ場。

**UI要素**
```
[キャラのアイコン + 見出し]
  「取材メモが届きました」（ミントの場合は「ミントからメモが届きました」）

[整理テキスト表示]

  セクション1: 引き出せた価値（箇条書き 3〜5点）
    「・〇〇（具体的に引き出した内容）」

  セクション2: 記事にできそうなテーマ（箇条書き 3点）
    「・〇〇というテーマで記事が書けます」

  セクション3: インタビューのやり取り（折りたたみ）
    「やり取りを見る ▼」で展開

[アクションボタン]
  「記事を作る」ボタン（大・目立つ）
  「もう少し話す」ボタン（インタビュー画面に戻る）
  「あとで記事にする」リンク（ダッシュボードに戻る、データは保存済み）
```

**設計注意点**
- この画面が一次情報整理の見せ場。「こんなに価値があった」と感じさせる
- 箇条書きは 3〜5点に絞る。多すぎると何が大事か分からない
- インタビュー原文は畳んでおく（圧迫感を減らす）

---

### 画面11: 記事生成・プレビュー `/projects/[id]/article`

**役割**
通常記事またはインタビュー風記事を生成してプレビュー・エクスポート。

**UI要素**
```
[タブ（上部）]
  「通常記事」タブ | 「インタビュー風記事」タブ

--- 通常記事タブ ---
[生成中状態]
  [レインのアイコン] 「記事を書いています...」

[完了状態]
  - タイトル（編集不可・MVPでは）
  - 記事本文（見出し付きの読み物形式）
  - エクスポートボタン群:
      「コピーする」（クリップボードにコピー）
      「Markdownでダウンロード」（.md ファイル）
  - 「再生成する」ボタン（別バリエーション）

--- インタビュー風記事タブ ---
[生成中状態]
  [ミントのアイコン] 「インタビュー記事を書いています...」

[完了状態]
  - タイトル
  - 導入文
  - 「この記事でわかること」（箇条書き）
  - インタビュアー紹介（キャラアイコン + 名前 + 一言）
  - 会話ブロック:
      ┌──────────────────────────────┐
      │ 回答者（ユーザー）       左寄り │
      │                  右寄り（キャラ）│
      └──────────────────────────────┘
      ※ 左=回答者、右=インタビュアー
  - エクスポートボタン群:
      「コピーする」
      「Markdownでダウンロード」
  - 「再生成する」ボタン

--- 共通 ---
[「このプロジェクトに保存する」ボタン]
[「インタビューからやり直す」リンク]
```

**設計注意点**
- 記事の編集機能はMVPでは省略。コピー/DLして自分のエディタで編集
- 「生成中...」は出さない。キャラアイコン + 「書いています」
- 再生成は同じインタビューデータから別バリエーションを作る

**データ**
- `articles` テーブル: 生成された記事を保存
- `article_type`: `normal` / `interview`

## DB テーブル設計（MVP最小構成）

```sql
-- ユーザー（Supabase Auth の auth.users を参照）

-- プロジェクト
projects
  id, user_id, name, hp_url, status, created_at, updated_at
  status: 'analyzing' | 'report_ready' | 'interview_ready' | 'interview_done' | 'article_ready'

-- 競合URL
competitors
  id, project_id, url, created_at

-- HP分析結果
hp_audits
  id, project_id, current_content, strengths, gaps, suggested_themes, raw_data, created_at

-- 競合分析結果
competitor_analyses
  id, project_id, competitor_id, gaps, advantages, raw_data, created_at

-- インタビューセッション
interviews
  id, project_id, interviewer_type, status, summary, themes, created_at, updated_at
  interviewer_type: 'mint' | 'claus' | 'rain'
  status: 'in_progress' | 'completed'

-- インタビューメッセージ
interview_messages
  id, interview_id, role, content, created_at
  role: 'user' | 'interviewer'

-- 生成記事
articles
  id, project_id, interview_id, article_type, title, content, created_at
  article_type: 'normal' | 'interview'
```

**RLS（Row Level Security）**
- 全テーブルに `user_id` または親テーブル経由のユーザー検証を設定
- 他のユーザーのデータは一切見えない

## 通知設計（メール）

## 画面遷移まとめ

```
[LP /]
  ↓ 「試してみる」
[登録 /auth/signup]
  ↓ 登録完了
[ダッシュボード /dashboard] ← ログイン後はここが起点
  ↓ 「新しいプロジェクトを始める」
[プロジェクト作成 /projects/new]
  ↓ URL登録 → 非同期で調査開始
[競合URL登録 /projects/[id]/competitors]
  ↓ 「調査を始める」（またはスキップ）
[調査・比較レポート /projects/[id]/report]
  調査中: ローディング表示 + メール通知予告
  調査完了: レポート表示
  ↓ 「インタビューを始める」
[インタビュアー選択 /projects/[id]/interviewer]
  ↓ 「来てもらう」
[インタビュー /projects/[id]/interview]
  ↓ AI判断 or ユーザー終了 → 確認
[インタビュー整理結果 /projects/[id]/summary]
  ↓ 「記事を作る」
[記事生成・プレビュー /projects/[id]/article]
  ↓ 「保存する」
[ダッシュボード /dashboard]（プロジェクト更新済み）

--- 再訪問 ---
[ダッシュボード]
  ↓ 既存プロジェクト「続ける」→ ステータスに応じた画面へ
  調査中 → /projects/[id]/report（ローディング状態）
  インタビュー待ち → /projects/[id]/interviewer
  インタビュー済み → /projects/[id]/summary
  記事生成済み → /projects/[id]/article

--- 離脱と再開 ---
どのステップでも離脱可能。
ダッシュボードに戻ってもデータは保存済み。
「続ける」で離脱した画面から再開できる。
```

## MVPで後回しにする機能

| 機能 | 理由 |
|---|---|
| 購入型インタビュアー（ハル/モグロ/コッコ） | 無料3体で価値検証を先に行う |
| 記事のインライン編集 | コピー/DLで自分のエディタで編集で十分 |
| Stripe 課金・プラン選択画面 | 価値を届けることが先 |
| メール以外の通知（Slack等） | メールで十分 |
| 管理者画面 | Supabase ダッシュボードで代替 |
| 分析ダッシュボード | Vercel Analytics で十分 |
| ソーシャルログイン | メール認証で十分 |
| 記事のSEO設定（メタ情報） | コピー先のCMSで設定 |
| モバイル最適化 | PC前提で始める |
| 多言語対応 | 日本語のみ |

---

## 実装順序

### Step 1: 認証 + ダッシュボードの骨格
- Supabase Auth セットアップ
- 登録・ログイン・ログアウト
- ダッシュボード（空状態）
- ルーティング保護（未認証はLPへリダイレクト）
- `projects` テーブル作成

**完了条件**: 自分たちがアカウントを作ってログインできる

---

### Step 2: プロジェクト作成 + 競合URL登録
- プロジェクト作成画面
- 競合URL登録画面
- `competitors` テーブル作成
- ダッシュボードにプロジェクトカード表示

**完了条件**: プロジェクトを作ってダッシュボードに表示できる

---

### Step 3: 調査・比較 AI + レポート画面
- HP調査 AI 実装（arch と協働）
- 競合比較 AI 実装
- Supabase Edge Functions でバックグラウンド処理
- `hp_audits` `competitor_analyses` テーブル作成
- ステータス管理（`analyzing` → `report_ready`）
- メール通知実装（Resend）
- レポート画面の表示

**完了条件**: URLを入れると調査結果が出てメールが届く

---

### Step 4: インタビュアー選択 + インタビュー画面
- インタビュアー選択画面
- インタビュー AI 実装（3キャラのプロンプト設計・arch 主導）
- 「充分な回答が得られたか」の AI 判断ロジック（arch）
- チャット UI 実装（リアルタイム表示）
- `interviews` `interview_messages` テーブル作成

**完了条件**: キャラを選んでインタビューが自然に終了できる

---

### Step 5: 整理結果 + 記事生成
- 整理 AI 実装
- 通常記事生成 AI 実装
- インタビュー風記事生成 AI 実装
- 会話ブロック UI（左=回答者、右=インタビュアー）
- クリップボードコピー機能
- Markdown ダウンロード機能（.md ファイル）
- `articles` テーブル作成

**完了条件**: 登録 → 調査 → インタビュー → 記事生成が一本通る

---

### Step 6: ランディングページ + 全体UX調整
- ランディングページ実装
- 各画面のキャラアイコン・世界観の実装
- エラーハンドリング・ローディング状態の整備
- 自分たちで実際に使ってみて修正

**完了条件**: 自分たちがこのツールで自社ブログ記事を1本作れる

---

### Step 7: P2画面の追加
- プロジェクト詳細画面
- 記事一覧・管理画面
- アカウント設定画面

**完了条件**: リピート使用に耐える状態

---

## この仕様書の扱い

- 実装着手前に build と arch が読む
- build: Step 1 から順に実装。各 Step 完了時に review にレビュー依頼
- arch: Step 3 の調査AI設計、Step 4 のインタビューAI設計（充分判断ロジック含む）、Step 5 の整理・記事生成AI設計を担当
- 変更があれば lead が `docs/decisions/` に記録してからこの仕様書を更新
