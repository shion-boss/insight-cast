# Task Breakdown

## 目的
依頼内容を、実装・確認・記録まで含めた実行可能なタスクに分解する。

この skill は、以下のような時に使う。

- 新機能を作る前
- 仕様が曖昧で、何から始めるべきか整理したい時
- 1つの要望が複数の作業にまたがっている時
- 「いま何をやるべきか」を明確にしたい時

## 前提
Insight Cast では、以下を守る。

- いきなり過剰実装しない
- まず MVP として成立する最小単位に落とす
- 実装前に、責務・画面・保存データ・受け入れ条件を確認する
- 人間の意思決定を置き換えない。比較案と推奨案までを出す
- 今回やることと、今回やらないことを分ける

## 出力ルール
タスク分解を行う時は、以下の順で整理する。

### 1. 依頼の要約
まず、依頼内容を1〜3文で要約する。

### 2. 目的
この作業の目的を明確にする。
「何を作るか」ではなく「何を達成したいか」を書く。

### 3. 完了条件
完了とみなせる条件を箇条書きで書く。
可能なら、ユーザー目線で確認できる表現にする。

### 4. タスク一覧
タスクを、実行順に分解する。

以下の粒度を意識する。

- 1タスク = 1つの責務
- 依存関係が分かる
- 実装、UI、DB、テスト、ドキュメント更新を必要に応じて分ける

### 5. 依存関係
「先に必要なもの」があれば明記する。

### 6. 今回やらないこと
スコープ外のことを明記する。
MVP外や将来対応のものをここに入れる。

### 7. 推奨実行順
最後に、着手順を短く提案する。

## 出力フォーマット

以下の形式で出力する。

### 依頼の要約
...

### 目的
...

### 完了条件
- ...
- ...

### タスク一覧
1. ...
2. ...
3. ...

### 依存関係
- ...
- ...

### 今回やらないこと
- ...
- ...

### 推奨実行順
1. ...
2. ...
3. ...

## 注意
- いきなりコードを書かない
- タスクが大きすぎる場合はさらに分割する
- 仕様が不足している場合は、不足点を明示する
- 「なんとなく必要そう」ではなく、目的に必要なものだけを含める
