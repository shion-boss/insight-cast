# Incident Triage

## 目的
障害の見立てを揃え、P0/P1/P2 の切り分けをぶらさない。

## 使う場面

- バグ報告を受けた直後
- 顧客問い合わせが障害起因か切り分けたい時
- 修正依頼を `エンジニア` / `AIデザイナー` に渡す時

## 見ること

### 優先度
- `P0`: 全停止、データ損失、セキュリティ事故
- `P1`: 主要導線が壊れている、同種エラー急増
- `P2`: 局所バグ、回避策あり、UX劣化

### 影響
- だれが困るか
- どの画面や API か
- 再現性があるか

### 次アクション
- 即時エスカレーションが必要か
- `エンジニア` と `AIデザイナー` のどちらに渡すか
- `review` の再確認が必要か

## 出力フォーマット

```md
## Triage

### 優先度
- P?

### 影響範囲
- ...

### 再現条件
- ...

### 仮説
- ...

### 次アクション
1. ...
2. ...
```

## 注意

- 原因不明のまま終えない
- 影響範囲が広い時は先に `lead` へ上げる
- 顧客返信文と技術調査メモを混ぜない
