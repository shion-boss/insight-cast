# cast-talk — 設計概要

## このスキルは何か

Insight Cast の AIキャスト同士が会話し、その内容をブログ記事として公開するコンテンツ自動生成パイプライン。毎日1本、自動的にテーマを選んで会話を作り、下書きとして保存する。

ドッグフーディング戦略（自社HPをこのツールで育てる）の中核コンテンツ生成機能。キャストが「本当に社員として働いている」世界観を、記事という形で外に見せる装置でもある。

---

## パイプライン構成

```
[1] テーマ選定
    入力: 既存記事タイトル一覧 + 過去 cast-talk タイトル一覧
    モデル: claude-haiku-4-5-20251001
    出力: { theme, format, interviewer, guest, reason }

       ↓

[2] 会話生成
    入力: theme, format, interviewer, guest
    モデル: claude-sonnet-4-6
    出力: { title, messages[], summary }

       ↓

[3] 下書き保存（build の仕事）
    Supabase に draft として保存
    担当キャストのアイコンを記事ヘッダーに添える
```

---

## モデル選定案

### [1] テーマ選定: claude-haiku-4-5-20251001

- タスクの性質: リストを読んで1件の JSON を返す。構造的な判断であり、深い推論は不要
- 毎日実行するため、コスト圧縮を優先
- 判断軸は「足りない観点」「キャラの専門性」「読者層の関心」という明確な基準があるので、haiku でも十分な精度が出る見込み
- 代替案: sonnet（より多様なテーマ提案が期待できるが、コストが上がる）
- **最終承認待ち（お金絡み）**

### [2] 会話生成: claude-sonnet-4-6

- タスクの性質: キャラの人格・口調・専門性を維持しながら、自然な会話を8〜14ターン作る
- 人格表現の質が記事の価値に直結するため、品質優先
- ここで世界観が崩れると記事全体が壊れる
- 代替案: haiku（コスト削減できるが、キャラの人格の一貫性と会話の自然さが落ちる可能性がある）
- **最終承認待ち（お金絡み）**

## パラメータ設計

### テーマ選定
```
temperature: 0.7
max_tokens: 400
```
- 温度はやや高めにして、毎日同じようなテーマが出るのを防ぐ

### 会話生成
```
temperature: 0.85
max_tokens: 3000
```
- 会話の自然さと多様性のために温度をやや高めに設定
- 8〜14ターン × 平均100文字 = 約1000〜1400文字。余裕を持って 3000 に設定

---

## 出力構造の設計方針

会話生成の出力を JSON にする理由:
- `messages[]` を build が UI に流し込みやすい形にする（左=回答者、右=インタビュアー）
- `summary` は記事のメタディスクリプションや SNS 投稿にそのまま使えるようにする
- `title` は自動命名。人間が確認して直せるように下書き状態で保存

---

## 世界観ガード

cast-talk の記事はInsight Cast自体の世界観を外に見せる場所。以下を必ず守る:

- キャストは「本当に対話している」体で書く。「AIが会話を作りました」という痕跡を出さない
- 記事の冒頭・末尾にキャストのアイコンを添える（build に依頼）
- 「生成」「処理」「モデル」などの語は会話内に出さない
- 40-60代経営者が読んで「自分の仕事に関係する話だ」と感じる内容にする

- レビュー対象: `theme-selection-prompt.md`, `conversation-prompt.md`, `characters.md`
- 特に見てほしい観点:
  - 各キャストの口調が既存の `interviewer-prompts/` と一貫しているか
  - 世界観語彙（「生成」「処理」「AIが」等）が会話出力に漏れていないか

  - format 別（interview / dialogue）で会話の質が変わっていないか

  - システムプロンプト内の「AIであることを強調する表現」は全除去済み
  - 「会社の従業員」として振る舞う設定を全キャストに記載済み
  - 禁止語彙リストを全プロンプトに明示済み
