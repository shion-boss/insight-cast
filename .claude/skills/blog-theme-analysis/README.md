# ブログ傾向分析 — 設計原則

## このディレクトリは何か

ユーザー（取材先）がすでに書いているブログ記事群を読み込み、「何を書いてきたか」「何が抜けているか」「インタビューで何を引き出せばブログが強化されるか」を特定するためのプロンプト設計。

---

## hp-analysis との違い

| 観点 | hp-analysis | blog-theme-analysis |
|---|---|---|
| 対象 | HP全体（トップページ・サービスページ・会社概要など） | ブログ記事群（複数本） |
| 見るもの | 事業の伝え方・構成・不足情報 | 書いてきたテーマの偏り・深さ・頻度 |
| 出力の用途 | インタビューの準備情報全般 | インタビューテーマの絞り込み |
| 位置づけ | 取材前の下地 | インタビュー設計の直前 |

hp-analysis が「この事業者は何者で、HPでどう伝えているか」を把握するのに対して、blog-theme-analysis は「この事業者はどんなことを発信してきたか、どこが手薄か」を把握する。両者は補完関係にある。

---

## 使用タイミング

```
hp-analysis（HP全体構造の把握）
    ↓
blog-theme-analysis（ブログ傾向の把握）  ← ここ
    ↓
インタビュー設計（テーマと担当キャストの決定）
    ↓
インタビュー実施
```

ブログが存在しない、または記事が3本未満の場合は本skillはスキップしてよい。その場合、hp-analysis の `suggested_themes` をインタビューテーマとして直接使う。

---

## なぜブログ分析がインタビューに必要か

ユーザーはすでにブログを書いているが、偏りがある。

- 得意なテーマや、書きやすいテーマに集中している
- 専門知識を紹介する記事は多いが、事業者本人の体験談や失敗談は少ない
- 書いているのに伝わっていないテーマがある（表面的すぎる・書き方が弱い）
- 業種的に重要なのに、当たり前すぎて誰も記事にしていないテーマがある

AIキャストがインタビューで聞くべきは「まだ書かれていない価値」にフォーカスすべきであり、すでに書かれているテーマを再度深掘りするのは効率が悪い。ブログの傾向を事前に掴むことで、インタビューの的が絞れる。

## 分析の3ステップ

### ステップ1: 傾向把握

複数のブログ記事テキストを受け取り、「何について、どのくらいの深さで、どのくらいの頻度で書いてきたか」を整理する。

**このステップで明らかにすること:**
- 頻出テーマ・カテゴリ（多く書いていること）
- 記事の深さの傾向（表面的な紹介 vs 体験・考え方の掘り下げ）
- 更新頻度・時系列の変化（何をきっかけに書き始め、何が続かなくなったか）
- 書き手の「得意な語り方」のパターン（事例紹介型・説明型・体験談型など）

**設計原則:**
傾向を「良い・悪い」で評価しない。記事の質を採点するのではなく、「書いている習慣とパターン」を地図として描くことが目的。評価が入ると、後段で「書くべきテーマ」ではなく「良い記事を増やすべきテーマ」という方向にずれる。

### ステップ2: 空白発見

傾向分析の結果と、hp-analysis の結果を組み合わせて、「書かれていない価値」を特定する。

**このステップで明らかにすること:**
- 書かれていないテーマ（業種的に重要だが記事化されていないもの）
- 書き始めているが浅いテーマ（もっと掘り下げられる余地があるもの）
- 競合が書いていてユーザーが書いていないテーマ（競合調査結果がある場合）

**設計原則:**
「書いていないこと」は必ずしも「書けないこと」ではない。書いていない理由は「当たり前すぎて気づいていない」「うまく言語化できていない」「書いていいか分からない」のいずれかが多い。空白の発見はそのまま「インタビューで引き出す価値の候補リスト」になる。

### ステップ3: インタビューテーマ接続

空白発見の結果と各キャストの特性を組み合わせて、「誰が・何を聞くか」の具体的な提案を出す。

**このステップで明らかにすること:**
- インタビューで引き出すべきテーマ（優先順位付き3〜5点）
- 各テーマに対してどのインタビュアーが向いているか
- インタビューで最初に聞くべき入り口の質問案

**設計原則:**
インタビューテーマは「何を書いてほしいか」ではなく「何を引き出せば事業者が知らない価値が出てくるか」の視点で設定する。事業者が「こんなこと聞かれると思わなかった」と感じる質問が、一次情報を引き出す。
