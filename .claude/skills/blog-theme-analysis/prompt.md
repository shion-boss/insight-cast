# ブログ傾向分析プロンプト

3つのプロンプトを順番に使う。各プロンプトの出力が次のプロンプトの入力になる。

---

## プロンプト1: ブログ傾向分析

### 用途

複数のブログ記事テキスト（またはURLから取得したMarkdown）を受け取り、書き手が何をどう書いてきたかの傾向を整理する。

### システムプロンプト

```
あなたは事業者のブログを読んで、発信の傾向とパターンを整理する担当です。
評価や採点は一切しないでください。「良い記事かどうか」を判断するのではなく、「何について、どのように書いてきたか」を地図として描くことが仕事です。
記事の書き方のくせ、よく登場するテーマ、書き方のパターンを客観的に読み取ってください。
```

### ユーザーメッセージ

```
以下は事業者のブログ記事です。複数の記事がある場合は、===記事区切り=== で区切られています。

## ブログ記事
${blogTexts}

## 整理してほしいこと

以下の観点で読み取れることを整理してください。
採点や優劣の判断はしないでください。「どんな傾向があるか」だけを書いてください。

出力形式（JSONのみ返してください）:
{
  "frequent_themes": [
    "よく書いているテーマや話題（3〜5項目）"
  ],
  "depth_pattern": "記事の深さの傾向（表面的な紹介が中心か、体験・考え方の掘り下げが多いか、その両方かを1〜2文で）",
  "update_pattern": "記事数と時系列の変化から読み取れること（記事が多い時期・少ない時期・止まっている時期など1〜2文で。日付情報がない場合は「確認できません」）",
  "writing_style": "書き手の語り方のパターン（事例紹介型・説明型・体験談型など、読んで感じた特徴を1〜2文で）",
  "notable_content": [
    "他の記事と比べて特に具体的・詳しく書かれている記事や話題（1〜3項目。特になければ空配列）"
  ]
}
```

### 設計意図

**「評価しない」という指示を明示する理由**

ブログ記事を分析させると、モデルは自然と「改善点」や「良かった点」を返しがちになる。しかしこのステップで必要なのは傾向の地図であり、採点ではない。「良い・悪い」の評価が入ると、次のステップで「良いテーマを増やす」という方向に引っ張られ、「まだ書かれていない価値を引き出す」という本来の目的からずれる。

**`depth_pattern` を自由文にする理由**

深さの傾向はカテゴリで切りにくい。「表面的な紹介が7割、体験談が3割」のような数値表現よりも、「商品説明は詳しいが、なぜその選択をしたかの理由は書かれていない」という自然文の方が、後段の空白発見で使いやすい。

**`notable_content` の目的**

特に詳しく書かれているテーマは、事業者が「得意な話題」または「最近力を入れていること」の手がかりになる。インタビューで「あの記事にあった〇〇の話、もう少し聞かせてください」という入り口に使える。

## プロンプト2: テーマ空白発見

### 用途

傾向分析の結果（プロンプト1の出力）と hp-analysis の結果を組み合わせて、「書かれていない価値」を特定する。競合調査結果がある場合はそれも加える。

### システムプロンプト

```
あなたは事業者のホームページとブログを読んで、まだ伝えられていない価値を見つける担当です。
「書いていないこと」は「書けないこと」ではありません。当たり前すぎて気づいていない、うまく言葉にできていない、書いていいか分からない、という場合がほとんどです。
見つけた空白は、インタビューで引き出せる価値の候補として扱ってください。
```

### ユーザーメッセージ

```
以下の情報を使って、この事業者がまだブログで伝えられていない価値を見つけてください。

## ブログ傾向分析の結果
${blogAnalysis}

## HP分析の結果
${hpAnalysis}

## 競合調査の結果（ある場合のみ）
${competitorAnalysis ?? "今回は使用しません"}

## 見つけてほしいこと

以下の3種類の「空白」に分けて整理してください。

出力形式（JSONのみ返してください）:
{
  "unwritten_themes": [
    {
      "theme": "テーマ名（短く）",
      "reason": "なぜ重要なのに書かれていないと判断したか（1〜2文）"
    }
  ],
  "shallow_themes": [
    {
      "theme": "テーマ名（短く）",
      "current": "現在どの程度書かれているか（1文）",
      "potential": "もっと深掘りできる余地（1文）"
    }
  ],
  "competitor_gaps": [
    {
      "theme": "テーマ名（短く）",
      "note": "競合は触れているがこの事業者は触れていない観点（1文）"
    }
  ]
}

補足:
- 競合調査結果がない場合は competitor_gaps は空配列にしてください
- 各カテゴリ2〜4項目を目安にしてください。無理に埋めなくてよいです
- テーマは事業者が日常会話で使う言葉で書いてください（「SEO」「コンバージョン」などの専門用語は避ける）
```

### 設計意図

**「当たり前すぎて気づいていない」という前提をシステムプロンプトに置く理由**

この文を入れないと、モデルは「欠けているもの」を「対策すべき欠点」として返す傾向がある。しかし事業者がブログに書いていない理由の大半は無自覚か言語化できていないためであり、欠点ではない。「インタビューで引き出せる」という前向きな枠組みをシステムプロンプトで設定しておく。

**3種類の空白に分ける理由**

- `unwritten_themes`: インタビューで一から引き出すもの。ゼロから掘る質問が必要
- `shallow_themes`: すでに話せる素地はあるが深さが足りない。「あの記事の続きを聞かせてください」という入り口が使える
- `competitor_gaps`: 競合と比べて言えていないことで、差別化の文脈で引き出せる

3種類を区別することで、インタビュアーがどのアプローチで聞くかを判断しやすくなる。

**「専門用語は避ける」指示の理由**

空白発見の出力は、次のステップでインタビューテーマに変換される。そのままキャストのプロンプトや事業者向けの画面にも使われる可能性があるため、事業者が日常で使う言葉で書いてもらう必要がある。

## プロンプト3: インタビューテーマ提案

### 用途

空白発見の結果（プロンプト2の出力）とキャラクター特性を受け取り、「誰が・何を聞くか」の具体的な提案を出す。

### システムプロンプト

```
あなたはインタビューのテーマ設計を担当しています。
事業者が「こんなこと聞かれると思わなかった」と感じる質問が、本人も気づいていない一次情報を引き出します。
「何を書いてほしいか」ではなく「何を引き出せば事業者が自分でも気づいていない価値が出てくるか」という視点でテーマを設定してください。
```

### ユーザーメッセージ

```
以下の情報をもとに、次のインタビューで取り上げるべきテーマを提案してください。

## ブログの空白発見の結果
${gapAnalysis}

## 担当インタビュアーの特性
${interviewerProfile}

担当インタビュアーの特性は以下のいずれかです（今回の担当: ${interviewerName}）:

- ミント（猫・女性）: お客様との関係、日常の気づかい、安心感を引き出す。雑談から入るのが得意
- クラウス（フクロウ・男性）: 業種の判断基準、技術的な違い、長年の経験から来る考え方を引き出す
- レイン（キツネ・男性）: 他社との違い、事業者がまだ言語化できていない「売り」、お客様が選ぶ理由を引き出す

## 提案してほしいこと

出力形式（JSONのみ返してください）:
{
  "interview_themes": [
    {
      "priority": 1,
      "theme": "インタビューテーマ（短く、事業者が日常で使う言葉で）",
      "why": "このテーマで話を聞くと何が引き出せるか（1〜2文）",
      "best_interviewer": "ミント / クラウス / レイン のどれが向いているか",
      "reason": "そのインタビュアーが向いている理由（1文）",
      "opening_question": "最初に聞く質問の案（1つ。「〜ですか？」という直球は避け、エピソードや状況が出やすい聞き方で）"
    }
  ],
  "session_note": "今回のインタビュー全体を通じて心がけるべきこと（1〜2文。事業者のタイプや書いてきた傾向を踏まえて）"
}

補足:
- interview_themes は優先度順に3〜5項目
- 1回のインタビューで深く扱えるテーマは1〜2つ。優先度1〜2を特に重点テーマとして設定してください
- opening_question は「最近〜な場面はありましたか」「〜と感じたのはどんな時でしたか」など、状況や感情から入る問いを心がけてください
```

### 設計意図

**`opening_question` の設計制約**

「最初の質問が会話の質を決める」という前提に立っている。直球の質問（「〜についてどう思いますか」）は事業者に「正しい答えを出さなければ」という構えを作る。「最近〜な場面はありましたか」「〜と感じたのはどんな時でしたか」という形式は、状況や感情を起点にするため、事業者が具体的なエピソードから話し始めやすい。

**`session_note` を設ける理由**

ブログの傾向を見ると、事業者のコミュニケーションスタイルがある程度推測できる。「事例紹介が得意な人」「説明的な書き方が多い人」「体験談を書く習慣がある人」では、インタビューの進め方も変わる。session_note はキャストのその日の会話のトーンを決める情報として使う。

**`best_interviewer` を各テーマに紐づける理由**

空白の種類によって、向いているキャストが異なる。
- 人との関係や気づかいに関する空白 → ミント
- 業種的な専門知識・判断基準の空白 → クラウス
- 差別化・競合との違いに関する空白 → レイン

事業者が「インタビュアーを選ぶ」行為に根拠を持たせるためにも、テーマとキャストの対応関係を明示する。

**優先度1〜2を重点テーマとして扱う指示の理由**

インタビューで3テーマ以上を均等に扱おうとすると、すべてが浅くなる。1回のインタビューは1〜2テーマを深く掘ることが目標。3〜5項目は「次回以降のインタビューの準備」としての意味もある。

## 各プロンプトの推奨モデルと温度設定

| プロンプト | 推奨モデル | temperature | 理由 |
|---|---|---|---|
| プロンプト1（傾向分析） | claude-sonnet | 0.3 | 客観的な整理が主。創造性より一貫性を重視 |
| プロンプト2（空白発見） | claude-sonnet | 0.5 | 傾向と実態を組み合わせる判断が必要。やや自由度を与える |
| プロンプト3（テーマ提案） | claude-sonnet | 0.7 | 質問案の生成に表現の幅が必要。固すぎると質問が定型化する |

**最終承認待ち（モデル選定はお金に関わるため人間の承認が必要）**

代替案: プロンプト1はコスト削減のため claude-haiku も候補。傾向の整理は定型処理に近いため。プロンプト2・3はインタビューの質に直結するため sonnet を推奨。

---

## 変数の補足

| 変数名 | 内容 | 取得元 |
|---|---|---|
| `${blogTexts}` | ブログ記事テキスト（複数本を `===記事区切り===` で連結） | Firecrawl等でクロール取得 |
| `${blogAnalysis}` | プロンプト1の出力JSON | プロンプト1の結果 |
| `${hpAnalysis}` | hp-analysis の出力JSON | `analyzeHp` 関数の結果 |
| `${competitorAnalysis}` | 競合調査の出力JSON（任意） | competitor-analysis の結果 |
| `${gapAnalysis}` | プロンプト2の出力JSON | プロンプト2の結果 |
| `${interviewerName}` | 選択されたキャラ名（ミント/クラウス/レイン） | ユーザーの選択 |
| `${interviewerProfile}` | 選択キャラの特性文（上記のプロンプト内の説明から該当部分） | キャラ定義から動的生成 |
