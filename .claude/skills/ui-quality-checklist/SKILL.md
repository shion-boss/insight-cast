# UI Quality Checklist

## 目的
「機能はあるが使いづらい」状態を防ぎ、普通に使える品質まで引き上げる。

この skill は、以下のような時に使う。

- 画面実装後の見直し
- UX改善フェーズ
- バグ修正ではなく使い勝手の磨き込みをしたい時
- review 観点で粗さを洗い出したい時

## 基本方針
Insight Cast では、UI品質を以下の基準で見る。

- 顧客が迷わない
- クリック可能なものが見て分かる
- 今どの状態か分かる
- 保存できたか分かる
- エラー時に次の行動が分かる
- 入力がストレスなく続けられる
- 世界観を壊さない
- 40〜60代のITリテラシーでも使いやすい
- 無言の状態を作らない
- 同じ役割のUIは同じ見え方に揃える

## デザイン実装の追加原則

- ボタンらしさは、動きではなく背景色・文字色・枠線の差で伝える
- 主操作 / 補助操作 / 低優先操作は、見た目だけで優先度が分かるようにする
- hover は最小限でよい。色・境界線・面の変化で十分なら動かさない
- クリックできないパネルや情報カードは、hover で浮かせたり動かしたりしない
- 影で立体感を作らない。余白・コントラスト・枠線・背景面の差で整理する
- 通常時の可読性を最優先する。hover 前の状態で読みにくい配色は避ける
- 同じ役割のUIは、ページをまたいでも同じ見た目・同じ文法に揃える
- 下層ページのヒーロー、CTA、セクション余白は共通パターンへ寄せる
- 読ませる面と押させる面を混同させない。押せるものだけが適切に目立つべき
- 迷ったら、派手さよりも「一目で分かること」を優先する

## 全画面共通チェック

### 操作可能要素
- クリック可能な要素は pointer になる
- hover 状態がある
- focus 状態がある
- disabled 状態がある
- 押せるものと押せないものが見分けられる
- 主要ボタンは通常時の時点で十分なコントラストがある
- 主操作と補助操作が見た目で区別できる
- 非ボタン要素がボタンのように過剰反応していない

### 状態表示
- ローディング中が分かる
- 保存中が分かる
- 成功時が分かる
- 失敗時が分かる
- 空状態が分かる
- 初期表示・空状態・失敗が混同されていない
- 無言で止まっているように見える時間がない
- 状態ごとに「次に何をすればいいか」がある

### データ表示
- 既存データが初期表示で反映される
- 保存後に最新状態へ更新される
- 初期値が空なのか読み込み失敗なのか区別できる
- 未保存変更があるか分かる
- 保存済みか、まだ編集中か分かる

### 導線
- 次に何をすればいいか分かる
- 重要なCTAが埋もれていない
- 戻る / キャンセル / 保存 の役割が明確
- 導線の文言が「操作説明」だけでなく、お願いごとの温度感になっている
- 同じ役割の導線で文言や見た目がバラついていない

### アクセシビリティ / 基本操作
- ボタンとリンクの使い分けが適切
- キーボード操作ができる
- フォーカス移動が自然
- ラベルと入力欄の関係が分かる
- focus-visible が主要な操作要素に揃っている
- hover / focus / disabled の差が見た目でも分かる

## フォームチェック

- 初期状態が閲覧モードか編集モードか明確
- 閲覧状態と編集状態が分かれている
- 保存前に何を変えたか分かる
- 未保存変更がある状態を見失わない
- 保存ボタン押下時に保存中状態になる
- 保存成功時に画面へ反映される
- 保存失敗時に原因または再試行方法が分かる
- 二重送信が防止されている
- 必須項目が分かる
- 不正入力時の表示がある
- キャンセル時に編集前へ戻せる
- 変更がない時に保存ボタンを押させない、または不要と分かる
- 「保存中 / 保存済み / まだ未保存」が区別できる

## チャット / 会話UIチェック

- 初回表示で入力欄に autoFocus する
- 送信後も連続入力しやすい
- Enter で送信できる
- Shift+Enter で改行できる
- 送信中状態が分かる
- 最新メッセージへ自然にスクロールする
- 会話の左右や話者が分かりやすい
- 入力欄を毎回クリックしなくてよい
- 長文入力時も崩れない
- 送信中表示が二重に出ない
- 通信失敗時に入力内容を失わず再送しやすい
- 「考え中」「整理中」などの待ち時間が無言にならない

## ナビゲーション / リンクチェック

- ログアウトや設定などのリンクが pointer になる
- hover 時に反応がある
- 現在地が分かる
- 誤クリックしにくいサイズになっている
- destructive action は必要に応じて確認がある
- ログアウトなど重要操作は、押した後の反応が分かる
- 小さすぎるテキストリンクだけで重要操作を持たせていない

## 世界観チェック

- 冷たいSaaS画面になっていない
- 重要な場面でキャラアイコンが適切に関与している
- テキストは分かりやすく、言い換えすぎていない
- 世界観のために操作性を犠牲にしていない
- 設定や入力でも「お願いごと」の温度感がある
- 状態表示にキャラや会社の気配がある
- 共通UIパターン（ヘッダー、主要ボタン、入力欄、状態カード）が崩れていない
- 情報カードや紹介パネルは静かに読める面として設計されている
- CTAだけが適切に視線を集め、全要素が競合していない

## 注意
- 「動く」だけで合格にしない
- 1つの画面で、表示・編集・保存・失敗時まで確認する
- UIの粗さはバグと同等に扱う
- 迷ったら「ユーザーが普通に使えるか」で判断する
- 迷ったら「いま何が起きていて、次にどうすればいいか」が見える方を選ぶ
- 新しい見た目を足す前に、既存の共通パターンへ寄せられないか確認する
