# UI Quality Checklist

## 目的
「機能はあるが使いづらい」状態を防ぎ、普通に使える品質まで引き上げる。

この skill は、以下のような時に使う。

- 画面実装後の見直し
- UX改善フェーズ
- バグ修正ではなく使い勝手の磨き込みをしたい時
- reviewer 観点で粗さを洗い出したい時

## 基本方針
Insight Cast では、UI品質を以下の基準で見る。

- 顧客が迷わない
- クリック可能なものが見て分かる
- 今どの状態か分かる
- 保存できたか分かる
- エラー時に次の行動が分かる
- 入力がストレスなく続けられる
- 世界観を壊さない
- 40〜60代のITリテラシーでも使いやすい

## 全画面共通チェック

### 操作可能要素
- クリック可能な要素は pointer になる
- hover 状態がある
- focus 状態がある
- disabled 状態がある
- 押せるものと押せないものが見分けられる

### 状態表示
- ローディング中が分かる
- 保存中が分かる
- 成功時が分かる
- 失敗時が分かる
- 空状態が分かる

### データ表示
- 既存データが初期表示で反映される
- 保存後に最新状態へ更新される
- 初期値が空なのか読み込み失敗なのか区別できる

### 導線
- 次に何をすればいいか分かる
- 重要なCTAが埋もれていない
- 戻る / キャンセル / 保存 の役割が明確

### アクセシビリティ / 基本操作
- ボタンとリンクの使い分けが適切
- キーボード操作ができる
- フォーカス移動が自然
- ラベルと入力欄の関係が分かる

## フォームチェック

- 初期状態が閲覧モードか編集モードか明確
- 閲覧状態と編集状態が分かれている
- 保存前に何を変えたか分かる
- 保存ボタン押下時に保存中状態になる
- 保存成功時に画面へ反映される
- 保存失敗時に原因または再試行方法が分かる
- 二重送信が防止されている
- 必須項目が分かる
- 不正入力時の表示がある
- キャンセル時に編集前へ戻せる

## チャット / 会話UIチェック

- 初回表示で入力欄に autoFocus する
- 送信後も連続入力しやすい
- Enter で送信できる
- Shift+Enter で改行できる
- 送信中状態が分かる
- 最新メッセージへ自然にスクロールする
- 会話の左右や話者が分かりやすい
- 入力欄を毎回クリックしなくてよい
- 長文入力時も崩れない

## ナビゲーション / リンクチェック

- ログアウトや設定などのリンクが pointer になる
- hover 時に反応がある
- 現在地が分かる
- 誤クリックしにくいサイズになっている
- destructive action は必要に応じて確認がある

## 世界観チェック

- 冷たいSaaS画面になっていない
- 重要な場面でキャラアイコンが適切に関与している
- テキストは分かりやすく、言い換えすぎていない
- 世界観のために操作性を犠牲にしていない

## 注意
- 「動く」だけで合格にしない
- 1つの画面で、表示・編集・保存・失敗時まで確認する
- UIの粗さはバグと同等に扱う
- 迷ったら「ユーザーが普通に使えるか」で判断する
