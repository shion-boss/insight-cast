# Spec Template

## 目的
機能仕様や画面仕様を、毎回同じ型で整理し、抜け漏れを防ぐ。

この skill は、以下のような時に使う。

- 新しい画面を追加する前
- 機能追加前に仕様を固めたい時
- 実装前に責務を明確にしたい時
- Claude Code に作業を依頼する前に前提を揃えたい時

## 前提
Insight Cast の仕様では、以下を重視する。

- 顧客にとって分かりやすいこと
- 40〜60代の中小企業ユーザーでも迷いにくいこと
- 世界観を壊さないこと
- 過剰な設定や複雑な導線を避けること
- MVPでは、まず体験の核を通すこと
- 状態が無言にならないこと
- 同じ役割のUIは同じパターンに揃えること

## 仕様整理の観点
仕様を書く時は、以下を最低限含める。

### 1. 機能名
短く分かりやすい名前。

### 2. 概要
この機能が何をするものかを説明する。

### 3. 目的
なぜこの機能が必要かを書く。
ユーザー価値や業務価値を明確にする。

### 4. 対象ユーザー
誰が使う機能かを書く。

例:
- 未ログインユーザー
- 顧客
- 管理者
- 自分たち自身（ドッグフーディング利用）

### 5. 画面 / 機能の責務
この画面や機能が担当する範囲を書く。
逆に、ここでやらないことも必要なら明記する。

### 6. UI要素
必要な要素を列挙する。

例:
- 入力欄
- ボタン
- テーブル
- 状態表示
- エラー表示
- CTA

### 7. 状態
画面や機能が持つ状態を書く。

例:
- 初期表示
- ローディング
- 空状態
- 編集中
- 未保存
- 保存中
- 成功
- 失敗

可能なら以下も区別する。
- 既存データ表示中
- まだ読み込み中
- 読み込み失敗
- 送信中
- 再試行可能な失敗

### 8. 保存データ
何を保存・取得するのかを書く。

### 9. バリデーション / 制約
入力制約や条件を書く。

### 10. エラー時の挙動
失敗した時にどう見せるかを書く。

### 11. 受け入れ条件
完成とみなせる条件を書く。

## 注意
- UIの見た目だけでなく、状態遷移まで書く
- 保存データがあるなら、必ず何を読み込み・何を更新するかを書く
- MVPか将来機能かを意識する
- 曖昧な言葉は避ける
- 実装に入る前提として使える粒度まで具体化する
- 「空なのか失敗なのか分からない」状態を仕様に残さない
- CTA文言は、意味が明確な範囲でやわらかい温度感にする
- 既存の共通UIパターンを使うか、新設が必要かも意識する
