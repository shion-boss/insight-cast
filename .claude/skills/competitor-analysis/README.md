# 競合比較分析 — 設計原則

## このディレクトリは何か

ユーザーが登録した競合HPを、自社HPと比較分析するための観点とプロンプト設計。
`app/api/projects/[id]/analyze/route.ts` の `compareCompetitor` 関数で使われるプロンプトを、設計意図と合わせて文書化する。

---

## 競合比較で何を明らかにするか

競合比較の目的は「競合を評価すること」ではない。
**「自社が伝えていない価値を外部視点で発見すること」**が目的。

競合HPを見ることで、「業界標準として伝えるべきこと」と「自社だけが伝えられること」の両方が見えてくる。

---

## 2つの比較観点

### gaps（競合が伝えていて自社にないもの）

競合が発信していて自社HPにない情報。

この情報はインタビューで2つの使い方ができる:
1. 「競合はこういうことをHPに書いているが、御社はどうか」と聞く — 同じことを伝えていないだけで実際には対応している可能性がある
2. 「この領域について御社のやり方はどうか」と聞く — 競合との違いを引き出すきっかけになる

### advantages（自社が競合より詳しく伝えているもの）

自社HPが競合より充実している領域。

これは「すでに伝えられている強み」として確認できる。
インタビューでは「この点については競合より詳しく書かれていますが、なぜそこを大事にされているのか」と掘り下げることで、強みの根拠を引き出せる。

---

## 競合比較の限界と注意点

### 競合HPの情報が古い・薄い場合がある

特に地域密着の中小企業同士の比較では、競合HPも更新が止まっている場合がある。
「競合にない」という判断が「競合も伝えていないだけ」の可能性がある。

分析結果を「絶対的な評価」として使わず、「インタビューの問いかけのヒント」として扱う。

### 競合が複数いる場合の扱い

現行実装では競合1社ずつ個別に比較し、結果を別々のレコードとして保存している。
複数競合の比較結果を統合して「業界全体で見た自社の立ち位置」を出す処理は現状なし。

将来的には複数競合の分析を統合するレポートの設計が必要。

### 比較は「優劣」ではなく「違い」として伝える

競合分析の結果を事業者に見せるとき、「あなたのHPが競合より劣っている」という表現は禁止。
「競合はこういう見せ方をしていますが、御社はどうお考えですか」というフラットな見せ方にする。
