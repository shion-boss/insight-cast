# 競合比較プロンプト

## 現行プロンプト（`compareCompetitor` 関数内）

```
2つのビジネスのホームページを比較してください。日本語で回答してください。

## 自社HP
${mainMarkdown.slice(0, 4000)}

## 競合HP
${competitorMarkdown.slice(0, 4000)}

## 出力形式（JSONのみ返してください）
{
  "gaps": ["競合が伝えていて自社にないもの（2〜4項目）"],
  "advantages": ["自社が競合より詳しく伝えているもの（1〜3項目）"]
}
```

---

## 各項目の設計意図

### テキスト量をHP分析（8000文字）より少なくしている理由

compareCompetitor は2つのHPを同時に処理する。
自社 4000文字 + 競合 4000文字 = 合計 8000文字で、HP分析と同等の情報量を確保しつつトークン消費を抑える設計。

ただしこの配分は暫定的なもの。両方のHPが情報量の多いサイトの場合、重要な情報が切り捨てられる。

### `gaps` を 2〜4項目に設定している理由

多すぎると「自社に欠けているものがたくさんある」という誤ったメッセージになる。
重要度の高い不足だけに絞り、インタビューでの問いかけに使えるレベルにする。

### `advantages` を 1〜3項目にしている理由

「自社が優れている点がある」という事実を最低1点出すことで、事業者の自信につながるデータにする。
競合比較が「自社の悪いところ探し」になるのを防ぐバランス設計。

---

## 現行実装の限界と改善案

### 限界1: 競合ごとに独立した比較になっている

競合A・B・C をそれぞれ個別に比較するが、「3社と比べて自社に共通して足りないもの」は分からない。

**改善案**: 複数の競合比較結果を集約して「業界全体で見た自社の立ち位置」を出す統合分析を追加する。

### 限界2: テキスト4000文字の切り捨て

HPの重要な情報（事例・料金・よくある質問など）が後半ページにある場合、切り取られる。

**改善案**: Firecrawlで複数ページを取得し、タイトル・要約・サービス概要ページを優先して構造化する前処理を入れる。

### 限界3: システムプロンプトがない

HP分析と同様、役割設定がなく汎用的な比較になっている。

**改善案**: 「あなたはHPの比較から発信機会を見つける専門家です」という役割設定をシステムプロンプトに追加する。

## 改善版プロンプト（試案）

```
システムプロンプト:
あなたは2つのホームページを読み比べ、自社が競合に対して発信できていない価値や、自社が優れている点を発見する専門家です。
優劣をつけることが目的ではなく、「自社のどこをインタビューで深めると価値が出るか」を明らかにすることが目的です。

ユーザーメッセージ:
以下の2つのホームページを比較してください。日本語で回答してください。

## 自社HP
${mainMarkdown.slice(0, 4000)}

## 競合HP
${competitorMarkdown.slice(0, 4000)}

## 比較の観点
- 競合が伝えていて自社が伝えていないこと（自社が対応していない可能性と、伝え方が足りない可能性の両方があること）
- 自社がより詳しく・具体的に伝えていること
- 明らかな表現の違い（どちらが読み手に分かりやすいか）

## 出力形式（JSONのみ返してください）
{
  "gaps": ["競合が伝えていて自社にないもの（2〜4項目）"],
  "advantages": ["自社が競合より詳しく・具体的に伝えているもの（1〜3項目）"]
}
```
