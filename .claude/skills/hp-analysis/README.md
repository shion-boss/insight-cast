# HP分析 — 設計原則

## このディレクトリは何か

ユーザー（取材先）のHPをインタビュー前に分析するための観点とプロンプト設計。
`app/api/projects/[id]/analyze/route.ts` の `analyzeHp` 関数で使われるプロンプトを、設計意図と合わせて文書化する。

---

## なぜインタビュー前にHP分析をするか

インタビューの質は「準備」で決まる。

何も知らない状態でインタビューに臨んだ場合、会話は「まず事業の概要を聞く」段階に多くの時間を使ってしまう。
HPを事前に読んでおくことで、「HPに書いてあること」はすでに分かっている前提でインタビューが進む。
これにより「HPに書いていないこと」「伝わっていること」「伝わっていないこと」の差分に集中できる。

事業者にとっても、「自分のHPを読んできてくれた」という感覚が信頼感を生む。

---

## 分析で見つけるべき4つの観点

### 1. 現在伝えていること（current_content）

HPに現在書かれている情報を整理する。
ここは「書いてあること」の確認であり、評価ではない。

**目的**: インタビューで「それはHPに書いてありますね」「では書いていない部分を聞かせてください」と言えるようにする。

### 2. 強みとして見えるもの（strengths）

HPの表現から読み取れる、事業者の強みと思われる記述を抽出する。
事業者が意識して書いたものと、意図せず滲み出ているものの両方を含む。

**目的**: インタビューで「こういうことが強みとして見えましたが、実際どうですか」と確認・深掘りする起点にする。HPの自己評価と事業者の実感を照合する。

### 3. 伝えきれていないこと・足りないもの（gaps）

HPを読んで「このサービスを知らない人がこれを見たとき、何が分からないか」という視点から不足を洗い出す。

**目的**: インタビューで「ここが伝わっていないと思うんですが、実際はどうなんですか」と直接聞く。ギャップを埋める素材をインタビューで引き出す。

### 4. インタビューで深めたいテーマ（suggested_themes）

上記3つの分析から、インタビューで特に掘り下げると価値が高そうなテーマを提案する。

**目的**: AIキャストがインタビュー開始時にこのテーマを意識して質問を組み立てる。また事業者にとって「今日はこの観点で話す」という方向感を作る。

---

## 分析の限界と注意点

### HPの内容に引きずられすぎない

HPの内容を分析すると「書いてあること」で事業者を評価しがちになる。
しかし本当の価値は「まだ書かれていないこと」の中にある。
分析の目的は「ギャップを見つけること」であり、HPの良し悪しを採点することではない。

### 古いHPでも価値がある

更新が止まっているHPでも、創業当時の思いや事業の骨格が読み取れる場合がある。
「古い情報でも、事業者の価値観の手がかりになる」という前提で読む。
